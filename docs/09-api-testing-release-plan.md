# API、测试与上线计划

本文是 V1 开发依据文档，定义 API 分组、接口契约原则、错误码与幂等要求、测试矩阵、灰度试运营和上线前验证。

## 1. API 契约总原则

1. 后端是金额、状态、退款、结算、保证金和账务的唯一权威。
2. 前端不得提交可信金额。`sale_price_cents`、`paid_amount_cents`、`service_fee_cents`、`agent_income_cents`、退款金额、结算金额都必须由后端计算或校验。
3. 金额统一使用 `amount_cents`，服务费率使用 `service_fee_bps=50`。
4. 服务费按用户实付金额计算，按分四舍五入，最终金额写入订单快照。
5. 所有接口返回稳定状态枚举、错误码和可展示错误信息。
6. 代理端 API 必须从认证主体派生 `agent_id/shop_id`，不能信任前端传参。
7. 回调 API 必须验签，先落回调原文，再按幂等键处理业务。
8. 导出 API 必须校验后台角色权限并写审计。
9. 不得提供任何邀请奖励、团队业绩、代理等级、返佣比例、团队订单奖励 API。
10. 受控两级渠道 API 只允许平台供货 -> 一级渠道商 -> 二级渠道商 -> 消费者，不得提供三级渠道能力。
11. 小程序和 H5 必须共用同一套订单、支付、履约、退款、结算和账务 API。

## 2. API 分组

### 2.1 用户端 API

1. 店铺详情：读取店铺名称、Logo、公告、客服微信、营业状态；支持小程序和 H5。
2. 店铺商品列表：读取当前店铺可售商品。
3. 商品详情：读取售价、权益说明、履约方式、售后规则。
4. 创建订单：后端校验店铺、商品、代理、保证金、风控和价格。
5. 发起支付：返回微信支付参数。
6. 支付结果查询：读取订单支付和履约状态。
7. 订单列表/详情：用户只能访问自己的订单。
8. 权益详情：用户只能访问自己的权益。
9. 创建售后：提交原因、说明、凭证、申请退款金额。
10. 售后详情：查看处理状态和结果。

### 2.1.1 H5 用户端 API

1. H5 店铺详情。
2. H5 店铺商品列表。
3. H5 商品详情。
4. H5 登录/会话占位。
5. H5 创建订单。
6. H5 发起支付占位。
7. H5 订单详情。
8. H5 权益详情。
9. H5 售后申请。

H5 API 必须从登录态派生用户身份，不得信任前端传入 `user_id`。

### 2.2 代理端 API

1. 入驻申请提交和详情。
2. 审核状态查询。
3. 保证金账户和流水查询。
4. 店铺详情、保存、分享码。
5. 平台商品库查询。
6. 代理商品上架、下架、改价。
7. 自有商品提交和审核状态。
8. 代理订单列表和详情。
9. 收益、冻结金额、结算记录、追扣记录。
10. 一级渠道能力申请和状态查询。
11. 二级渠道商管理。
12. 转供商品和转供价配置。
13. 一级/二级渠道结算查询。

所有代理端 API 必须以后端认证主体派生 `agent_id/shop_id`。即使请求携带其他 `agent_id/shop_id`，也必须忽略或拒绝。

### 2.3 平台后台 API

1. 代理审核。
2. 保证金管理。
3. 平台商品库管理。
4. 代理商品审核。
5. 订单管理。
6. 履约管理。
7. 售后仲裁。
8. 退款审批。
9. 结算管理。
10. 人工打款回填。
11. 追扣处理。
12. 风控冻结和解冻。
13. 投诉处理。
14. 审计日志。
15. 基础看板。
16. 一级渠道授权审核。
17. 二级渠道审核。
18. 渠道关系管理。
19. 转供价审计。
20. 多渠道支付配置占位。
21. 平台自营店管理。
22. 平台自营商品上架。
23. 平台自营订单、履约、售后、退款。
24. 平台自营看板和对账。

### 2.4 回调 API

1. 支付回调。
2. 退款回调。
3. 履约回调。

要求：

1. 验签。
2. 校验金额或业务标识。
3. 写原始 payload。
4. 使用幂等键。
5. 业务处理失败时保留可重试状态。
6. 重复回调不得重复履约、重复退款、重复入账、重复写 ledger。

支付渠道字段必须能区分 `wechat_miniprogram`、`wechat_h5_jsapi`、`wechat_h5`、`alipay_wap`、`mock`。小程序内使用微信小程序支付；微信内 H5 使用微信 JSAPI；微信外手机浏览器 H5 使用微信 H5；普通手机浏览器支持支付宝 WAP。小程序内不设计支付宝支付入口。

### 2.5 导出 API

1. 结算单导出。
2. 人工打款文件导出。
3. 支付对账导出。
4. 退款对账导出。
5. 代理收益明细导出。
6. 保证金流水导出。
7. ledger 明细导出。
8. 审计日志导出。

导出动作必须写 `audit_logs`。

## 3. 错误码原则

错误码至少区分：

1. 参数错误。
2. 认证失败。
3. 权限不足。
4. 资源不存在。
5. 状态不允许。
6. 金额校验失败。
7. 低于最低限价。
8. 重复请求。
9. 风控限制。
10. 保证金不足。
11. 外部渠道失败。
12. 系统内部错误。

错误信息需要能支持前端展示，但不得泄露其他代理、后台财务或风控策略敏感信息。

## 4. 幂等与唯一约束

1. 支付回调：`pay:{channel}:{channel_trade_no}`。
2. 退款回调：`refund:{channel}:{channel_refund_no}`。
3. 履约尝试：`fulfill:{order_item_id}:{attempt_no}`。
4. 权益发放：`entitlement:{order_item_id}:{rights_code}`。
5. 结算生成：`settlement:{agent_id}:{period_start}:{period_end}:{batch_no}`。
6. 结算明细：单层订单数据库唯一 `settlement_items.order_id`；两级渠道订单数据库唯一 `settlement_items(order_id, settlement_role)`。
7. 追扣：`clawback:{source_type}:{source_id}:{agent_id}`。
8. 保证金交易：`deposit:{type}:{source_type}:{source_id}:{agent_id}`。
9. 审计事件：`audit:{request_id}`。

所有幂等失败重试必须可观测，不能静默吞错。

## 5. 验收测试总原则

1. 测试重点不是页面可用，而是交易、履约、售后、结算、风控、审计、权限隔离和账务一致性。
2. 金额一律校验 `amount_cents`，不得用浮点数作为断言口径。
3. 服务费按用户实付金额 `50 bps` 计算，必须覆盖整分取整、部分退款比例、全额退款、责任方承担。
4. 前端展示金额只能作为辅助，订单、退款、结算、追扣、保证金扣减必须以后端落库结果为准。
5. 支付、退款、履约、结算、追扣、保证金交易、导出任务必须验证幂等。
6. 代理端 API 必须从认证主体派生 `agent_id/shop_id`，测试必须覆盖篡改参数无效。
7. 禁止出现多级分销、下级返佣、团队奖、邀请奖励、代理等级体系等字段、页面、接口和统计口径。
8. 受控两级渠道测试必须验证二级不能继续创建三级。
9. H5 测试必须验证浏览器可访问、身份隔离和与小程序共用后端。

## 6. 验收测试矩阵

| 模块 | 关键场景 | 预期结果 | 必须校验数据/日志 |
| --- | --- | --- | --- |
| 代理入驻 | 提交申请、审核通过、审核拒绝、重复提交 | 未审核不能开店；拒绝记录原因；通过后进入保证金环节 | `agent_applications.status/reject_reason`、`agents.status`、`audit_logs` |
| 保证金 | 未缴纳销售、缴纳确认、扣减、余额不足限制 | 未缴纳不能正式销售；扣减有原因；不足触发限制销售/结算/打款 | `deposit_accounts`、`deposit_transactions`、`risk_freezes`、`audit_logs` |
| 店铺归属 | 配置店铺、客服微信、冻结/禁用店铺、分享路径进入 | 用户看到正确代理店铺；禁用店铺不能下单；客服微信为该店铺配置 | `shops.agent_id/status`、订单快照 `shop_id/agent_id` |
| H5 店铺 | 浏览器访问 H5 店铺、H5 商品详情、H5 下单 | H5 可在微信外浏览器打开；数据与小程序一致；用户端不展示供货价和结算信息 | H5 响应、`orders`、`order_amount_snapshots` |
| 平台自营店 | 平台配置自营店、自营商品、自营售价、用户下单 | 自营订单归属 `platform_self_operated`；不生成渠道结算；进入收入成本对账 | `shops.owner_type`、`orders.sales_channel_type`、`ledger_entries` |
| 商品/定价 | 平台商品创建、上架、下架、最低价校验、价格变更 | 低于最低限价失败；等于/高于成功；历史订单快照不变 | `platform_products`、`agent_products.sale_price_cents`、`order_amount_snapshots` |
| 两级渠道授权 | 一级申请、平台审核、暂停/关闭授权 | 未授权一级不能创建二级；授权变更有审计 | `channel_authorizations`、`audit_logs` |
| 二级渠道入驻 | 二级提交、审核、缴保证金、开店 | 未审核或未缴保证金不能销售；二级不能创建三级 | `channel_relations`、`agents.status`、`deposit_accounts` |
| 转供价 | 一级配置转供价、二级设置最终售价、低价绕过 | 转供价不得低于平台供货价；最终售价不得低于转供价和平台限制价 | `channel_product_offers`、`audit_logs` |
| 供货链快照 | 二级店铺下单、调价后查看历史订单 | 固化平台供货价、一级转供价、最终售价、一级差价、二级差价 | `order_amount_snapshots.pricing_snapshot_json` |
| 自有商品审核 | 代理提交、审核通过、拒绝、下架、风控限制 | 未审核不可售卖；拒绝有原因；下架后不可购买 | `agent_product_reviews.status/reject_reason`、`audit_logs` |
| 下单支付 | 从代理店铺下单、篡改价格、下架商品下单、未缴保证金下单 | 后端计算金额；订单归属正确；非法状态拒单 | `orders`、`order_items`、`order_amount_snapshots` |
| 支付回调幂等 | 成功回调、重复回调、延迟回调、金额不一致回调 | 订单只支付成功一次；不重复履约、不重复流水；金额不一致拒绝或异常挂起 | `payments`、`payment_callbacks.idempotency_key`、`ledger_entries` |
| 多渠道支付占位 | 小程序内微信支付、微信内 H5 JSAPI、微信外 H5、支付宝 WAP、mock 渠道创建支付单 | 统一支付模型；mock 不绕过业务状态机；小程序内不出现支付宝入口；真实渠道未开通时有明确待开通状态 | `payments.channel`、`payment_callbacks` |

| 售后退款 | 未履约全退、已履约未使用退款、已使用拒退、平台/代理/一级/二级/用户/混合责任 | 退款金额不超过实付；责任归属必填；退款中冻结结算 | `after_sales`、`refunds`、`orders.settlement_status` |
| 部分退款 | 多次部分退款、累计超过实付、服务费按比例退、责任拆分 | 累计退款不得超过实付；平台/代理/一级/二级承担准确；服务费口径一致 | `refunds`、`after_sales.platform_bear_cents/agent_bear_cents/first_tier_bear_cents/second_tier_bear_cents/service_fee_refund_cents`、`ledger_entries` |
| 两级退款追扣 | 平台责任、一级责任、二级责任、混合责任、已结算后退款 | 一级/二级承担金额可拆分；已结算后分别追扣；保证金不足触发限制 | `after_sales`、`clawbacks`、`ledger_entries`、`deposit_transactions` |
| 平台自营退款 | 自营未履约退款、已履约退款、支付通道费处理 | 自营退款不生成一级/二级追扣；进入平台退款、成本和 ledger 对账 | `after_sales`、`refunds`、`ledger_entries` |
| 结算 | T+1 可结算、未履约不结算、退款/投诉/风控冻结、重复生成结算单 | 只有符合条件订单进入结算；单层订单不能重复结算 | `settlement_sheets`、`settlement_items.order_id` 单层唯一 |
| 一级/二级结算 | 二级订单履约成功后分别生成一级和二级结算明细 | 一级差价和二级差价分别结算；同一角色同一订单不重复结算 | `settlement_items(order_id, settlement_role)` 唯一、`ledger_entries` |
| 人工打款 | 导出打款文件、财务确认打款、回填流水 | 结算单金额等于明细汇总；打款确认后状态正确 | `manual_payouts`、`settlement_sheets.status`、`audit_logs` |
| 追扣 | 已结算后退款、从未结算收益扣、从待打款扣、从保证金扣 | 追扣顺序清晰；不足时限制店铺/结算；代理端可见原因 | `clawbacks`、`ledger_entries`、`deposit_transactions` |
| 保证金扣减 | 投诉赔付、售后失责、追扣不足转保证金 | 扣减有业务原因、关联订单/事件、操作人 | `deposit_transactions`、`ledger_entries`、`audit_logs` |
| 风控冻结 | 冻结订单、解冻订单、冻结店铺、限制结算、商品下架 | 冻结期间不可结算/下单；解冻后按状态重新流转 | `risk_freezes`、`orders.risk_status`、`shops.status` |
| 权限隔离 | 用户访问他人订单、代理跨店查询、代理查看平台财务、前端传入他人 `agent_id` | 返回 403/404；无数据泄露；以后端认证主体为准 | API 响应、访问日志、`audit_logs`、数据行归属字段 |
| 后台 RBAC | 运营做财务打款、财务改商品、代理执行后台财务操作 | 无权限操作被拒绝；敏感操作有审计 | `admin_users`、`roles`、`permissions`、`audit_logs` |
| 审计日志 | 审核、冻结、退款仲裁、结算生成、人工打款、保证金扣减 | 操作人、时间、对象、前后状态、原因完整 | `audit_logs.operator_id/action/target_type/before/after/reason/ip` |
| 导出对账 | 订单明细、支付流水、退款流水、结算单、保证金流水、日报 | 明细汇总等于日报；支付/退款渠道流水一致 | 导出文件、`ledger_entries`、渠道流水 |
| 小程序提审 | 类目、支付能力、隐私协议、用户协议、售后规则、客服入口、分享路径 | 提审资料完整；虚拟商品内容与类目一致；路径可访问 | 小程序配置、协议页面、客服入口、支付配置、测试账号 |

## 7. 安全与越权专项测试

P0 必测：

1. 篡改 `shop_id/agent_id` 创建订单：后端忽略前端传参或拒绝，订单只能归属当前店铺上下文。
2. 篡改 `sale_price_cents/paid_amount_cents/service_fee_cents`：后端重新计算，非法金额拒单。
3. 代理 A 查询代理 B 的订单、客户、商品、结算、售后：返回 403/404。
4. 用户 A 查询用户 B 的订单和权益：返回 403/404。
5. 重复支付回调、重复退款回调、重复履约回调：不重复更新订单、不重复发权益、不重复写 ledger。
6. 绕过最低限价直接调用 API：保存失败，并记录安全事件。
7. 禁用/冻结店铺继续下单：下单失败，已有历史订单不改变归属。
8. 代理尝试执行后台财务操作、结算生成、人工打款、保证金扣减：必须拒绝。
9. API、数据库、页面、导出中不得出现分销层级、邀请奖励、团队奖、下级佣金、代理等级等功能痕迹。
10. 二级渠道尝试创建三级渠道必须失败并记录安全事件。
11. H5 篡改用户 ID、店铺 ID、渠道 ID 创建订单必须失败或以后端上下文为准。

## 8. 上线前全链路验证

上线前必须在灰度环境完成：

1. 至少 1 个代理完成入驻审核、保证金缴纳、店铺配置、客服微信绑定。
2. 至少 1 个平台商品完成创建、代理上架、代理售价高于或等于最低限价。
3. 至少 1 单用户从代理店铺进入并完成 mock 购买；账号开通后补真实购买。
4. 账号未开通阶段至少 1 单完成 mock 支付回调，并重复投递支付回调验证幂等；账号开通后再补真实支付回调验收。
5. 至少 1 单完成虚拟权益履约，用户端可查看权益凭证。
6. 至少 1 单 mock 退款，覆盖退款中冻结结算、退款回调幂等、账务流水；账号开通后补真实退款。
7. 至少 1 张 T+1 结算单，校验单层 `settlement_items.order_id` 不重复、两级 `settlement_items(order_id, settlement_role)` 不重复。
8. 至少 1 次人工打款确认，记录打款流水、操作人和时间。
9. 至少 1 单已结算后退款，生成追扣记录。
10. 至少 1 次保证金扣减，校验原因、关联事件、余额和审计日志。
11. 至少 1 次订单风控冻结/解冻，验证冻结期间不进入结算。
12. 至少 1 次账务日报平账：订单实付、支付渠道、退款、服务费、代理收益、冻结、结算、追扣、保证金余额全部一致。
13. 至少 1 个 H5 店铺可在微信外浏览器打开并完成 mock 下单。
14. 至少 1 个平台自营店完成小程序/H5 下单、履约、退款和对账。
15. 至少 1 个一级渠道商完成二级渠道授权，1 个二级渠道商完成入驻、缴保证金和二级店铺下单。
16. 至少 1 单两级渠道订单完成供货链快照、退款、追扣和一级/二级结算验证。

## 9. 灰度试运营计划

范围：

1. 3-5 个测试代理。
2. 5-10 个低风险平台虚拟商品。
3. 每个代理至少 1 单 mock 支付；真实支付账号开通后再做小额真实支付。
4. 至少覆盖 1 单 mock 退款、1 张结算单、1 次人工打款、1 次追扣、1 次保证金扣减、1 次风控冻结；账号开通后补真实退款。
5. 每日输出账务日报和平账结果。
6. 客服完成售前咨询、售后申请、平台仲裁话术演练。

通过标准：

1. 支付、履约、退款、结算、追扣、保证金扣减全链路无 P0/P1 缺陷。
2. 日报平账差异为 0；如有差异，必须能定位到具体订单/流水并修正。
3. 无越权访问、无跨代理数据泄露、无重复发放或重复入账。
4. 小程序提审资料、协议、隐私、售后规则、客服入口完整。
5. 未出现重大投诉、违规宣传、履约大面积失败或虚拟商品类目风险。

回滚或暂停标准：

1. 出现金额计算错误、重复入账、重复履约、结算重复、退款超额任一问题。
2. 出现代理或用户越权访问。
3. 支付或退款渠道流水无法对账。
4. 风控冻结失效，异常订单进入结算。
5. 小程序审核、支付能力、类目合规存在明确阻断。
6. 出现多级分销、邀请奖励、团队返佣等违规功能或文案。
7. 二级渠道能继续创建三级渠道。
8. H5 和小程序订单数据不一致或出现用户越权。
9. 平台自营订单错误生成渠道结算或暴露给渠道商。

## 10. 发布阶段

### 阶段 0：文档、原型和口径定稿

完成系统架构、数据库、API、状态机、权限、测试矩阵和原型。财务口径未最终定稿的，必须以系统配置项和默认值方式记录。

### 阶段 1：数据库与后端主链路

优先实现核心实体、订单快照、支付回调、履约、售后、结算、保证金、ledger、RBAC 和审计。

### 阶段 2：用户端交易闭环

实现代理店铺、商品、下单、支付、订单、权益、售后。

### 阶段 3：代理端经营闭环

实现入驻、保证金、店铺配置、选品定价、自有商品、订单收益、结算和追扣展示。

### 阶段 4：平台后台、财务和风控闭环

实现代理审核、商品审核、订单、履约、售后仲裁、结算单、人工打款、风控冻结、审计、对账导出。

### 阶段 5：联调、灰度和提审

完成安全越权测试、幂等测试、账务日报平账、mock 支付退款结算灰度、小程序提审材料；真实支付账号开通后再补真实支付退款灰度。

### 阶段 6：H5 与两级渠道灰度

完成 H5 浏览器访问、H5 用户身份隔离、平台自营下单与对账、一级渠道授权、二级渠道入驻、转供价、供货链快照、一级/二级结算和两级退款追扣验证。

### 阶段 7：多渠道支付与服务器上线

完成微信小程序支付、微信内 H5 JSAPI、微信外 H5 支付、支付宝手机网站支付的配置接入计划；完成 API 域名、H5 域名、后台域名、HTTPS、备案、小程序合法域名、PostgreSQL、Redis、对象存储、日志、备份和监控检查。

## 11. 待确认参数对测试和上线的影响

1. 保证金金额、缴纳方式、退还观察期未定：影响保证金缴纳、扣减、退还、余额不足限制用例断言。
2. 服务费退款口径未完全定稿：影响平台责任、代理责任、用户原因、部分退款时服务费承担方和金额断言。
3. T+1 起算点未最终确认：影响结算可结算时间断言；V1 默认按履约成功时间。
4. 店铺绑定期是否 30 天未定：影响用户多入口进入、归属覆盖、直接访问平台首页下单测试。
5. 自有商品权限范围未定：影响哪些代理可提交、销售额度、保证金约束、最低价校验用例。
6. 履约方式和权益是否可撤销未定：影响自动发放、人工发放、重试、撤销、退款后权益状态测试。
7. 已使用/部分使用权益退款规则未定：影响部分退款、拒退、人工仲裁和责任拆分测试。
8. 客服升级时限未定：影响售后超时升级和平台介入用例。
9. 是否设置最高售价或价格异常提醒未定：影响价格风控测试。
10. 小程序主体、类目、虚拟商品支付能力未确认：影响提审通过率和真实支付灰度计划。
11. H5 登录方式未确认：影响 H5 订单身份、售后和权益隔离测试。
12. 一级渠道授权条件未确认：影响二级渠道能力开通测试。
13. 二级保证金金额未确认：影响二级销售限制和追扣用例。
14. 服务费在一级/二级之间的承担口径未确认：影响两级结算断言。
15. 支付渠道接入顺序未确认：影响真实支付灰度计划。
