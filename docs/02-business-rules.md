# 业务规则与账务规则

## 1. 商业模式

项目采用 B2B2C 微商小店/受控商户供货模式。

平台向商户提供虚拟商品供货和履约能力，也可以开设平台自营店直接销售。商户在自己的小店中销售商品，用户看到的是商户店铺和商户售价。单层店铺收益来自售价与平台供货价之间的差价；平台服务费由后台配置决定，可以开启、关闭，也可以调整比例。

禁止将本项目设计为返佣分销、下级返佣或团队业绩奖励系统。当前允许平台自营和受控三级以内商户价差供货并存：平台自营店 -> 消费者；平台供货 -> 一级商户 -> 二级商户 -> 三级商户 -> 消费者。一级商户收益来自一级转供价与平台供货价之间的供货价差，二级商户收益来自二级转供价或最终售价与一级转供价之间的价差，三级商户收益来自最终售价与二级转供价之间的销售价差。

## 2. 商品规则

### 2.1 平台商品

平台商品由平台创建和维护。

每个商品至少包含：

1. 商品名称。
2. 商品分类。
3. 商品详情。
4. 权益说明。
5. 平台供货价。
6. 最低销售价。
7. 建议销售价。
8. 履约方式。
9. 售后规则。
10. 上架状态。

平台商品由平台负责主要履约和供货质量。

平台商品、商户价格和转供保护规则：

1. 平台商品资料可以沿供货链开放给下级用于选品和销售，但各级后台只能看到自己这一层需要看到的商户价格。
2. 平台给一级商户的供货价只对平台和对应一级商户可见，二级/三级商户绝对不能看到平台给一级的供货价。
3. 一级商户选择平台商品并点击上架后，商品自动进入该一级商户自己的小店；一级商户可设置自己的前台销售价。
4. 一级商户可以把自己已选择的商品开放给二级商户，并设置给二级商户的一级转供价。
5. 二级商户登录后可以看到一级商户开放给自己的商品资料和一级转供价，但看不到平台供货价、平台给一级的供货价或一级真实进货成本。
6. 二级商户选择一级商户开放的商品并点击上架后，商品自动进入该二级商户自己的小店；二级商户可设置自己的前台销售价。
7. 如启用三级，二级商户可以把自己已选择的商品开放给三级商户，并设置给三级商户的二级转供价；三级只能看到二级给自己的二级转供价，不能看到一级转供价和平台供货价。
8. 平台后台可查看完整供货链和各级价格；各级商户后台只能看到自己业务所需的上游价格、自己的售价和自己的收益，不得越级查看上游真实成本。
9. 商品资料、商品详情、商品图片、供货价、转供价、售价、库存和卡密必须来自数据库/API 或对象存储；生产运行代码不得硬编码。

### 2.1.1 代理商品展示覆盖规则

平台商品源只由平台维护。商户代理平台商品后，可以为自己的店铺单独修改展示内容，但不能改变商品来源、库存来源、供货链和结算归属。

规则：

1. 商户可以只改售价，也可以改自己店铺里的商品展示名、副标题、详情、使用说明、图片、分类、标签、规格和详情分段。
2. 商户展示修改只影响该商户自己的小店和该商户卖出的订单快照，不会回写平台商品库。
3. 下级选品时看到直接上级的展示版本。一级改了展示，二级选品看到一级版本；二级再改展示，三级选品看到二级版本。
4. 买家看到的是销售商户自己的展示版本和销售价。
5. 订单快照使用销售商户当时的展示版本，同时保留平台商品源和供货链字段，方便履约、结算、售后和审计。
6. 不管商户怎么改展示，平台商品仍然属于平台供货；平台码池仍由平台负责，商户不能查看上游卡密明文。
7. 三级商户卖出平台代理商品时，库存从平台或上游配置的主码池锁定和发放；M1、M2、M3 不需要各自复制一份平台商品库存。
8. 展示覆盖不是“把平台商品变成商户自有商品”。商户自有商品必须走自有商品审核和自己的库存/履约规则。

### 2.2 商户自有商品

商户可以上传自有虚拟商品，但必须经过平台审核后才能售卖。

商户自有商品建议规则：

1. 商户负责主要履约。
2. 商户负责主要售后。
3. 平台保留审核、下架、冻结和仲裁权。
4. 商户自有商品造成用户投诉或损失时，优先从商户收益或保证金中扣减。

第一版支持商户提交自有虚拟商品，所有自有商品必须经过平台审核后才可在该商户店铺售卖。平台可以基于风控策略限制部分商户使用该能力，但系统能力需要具备。

### 2.3 平台自营商品

平台可以将商品上架到平台自营店直接销售。平台自营商品由平台负责定价、履约、售后和对账。

平台自营规则：

1. 平台自营店不属于一级商户。
2. 平台自营订单不参与商户供货链。
3. 平台自营订单不产生商户供货差价，不生成商户供货结算。
4. 平台自营订单必须记录最终售价、履约成本、支付手续费、平台自营毛收益和订单快照。
5. 平台自营退款由平台承担相应履约成本、退款损失和通道费处理。

## 3. 定价规则

商户可以自由设置商品销售价，但不得低于平台配置的最低销售价。

后台字段建议：

1. 平台供货价：平台与商户之间的结算基础。
2. 最低销售价：商户售价不得低于该价格。
3. 建议销售价：平台给商户的推荐销售价格。
4. 商户销售价：用户实际看到和支付的价格。
5. 一级转供价：一级商户给二级商户的供货价。
6. 二级转供价：二级商户给三级商户的供货价。
7. 最终销售价：消费者实际看到和支付的价格。

价格校验必须以后端为准，前端提示只能作为辅助。

三级以内商户供货价格校验：

1. 一级转供价不得低于平台供货价。
2. 二级转供价不得低于一级转供价。
3. 最终销售价不得低于销售商户的上游供货价和平台最低销售价。
4. 三级商户不得配置下级转供价或创建四级商户。
5. 平台供货价、一级转供价、二级转供价、最终销售价必须写入订单供货链快照。

示例：

```text
平台供货价：100.00，仅平台与一级可见
一级给二级的转供价：110.00，仅一级、该二级和平台可见
二级最终销售价：150.00，消费者可见
一级价差：110.00 - 100.00 = 10.00
二级销售价差：150.00 - 110.00 = 40.00
```

这不是返佣，也不是按比例分成；它是上游供货价和下游售价之间的真实经销价差。

后台展示要求：

1. 一级商户后台可看到平台给自己的供货价、自己给二级配置的一级转供价、自己的前台售价和自己的价差。
2. 二级商户后台只能看到一级给自己的一级转供价、自己给三级配置的二级转供价、自己的前台售价和自己的价差。
3. 三级商户后台只能看到二级给自己的二级转供价、自己的前台售价和自己的价差。
4. 二级/三级后台、导出、接口响应、浏览器网络返回值中都不能出现平台给一级的供货价。
5. 后台表格、快捷操作、导出和前台页面不得使用固定测试 ID、固定测试价格、固定测试商品、固定收款码或固定客服二维码。
6. 平台账户可以看到所有转供价、各级商户销售价，以及各级商户自己上架的商品。

## 4. 订单归属规则

用户通过商户店铺邀请码进入 H5 店铺后，下单订单归属于该商户店铺。

建议第一版采用以下归属规则：

1. 用户从商户店铺进入时，当前会话默认绑定该商户店铺。
2. 用户在该店铺内下单，订单归属该销售商户。
3. 若用户直接进入平台首页，默认进入平台主店，除非仍在有效店铺绑定期内。
4. 是否设置 30 天店铺绑定期需要业务方最终确认。

订单归属必须在下单时固化，后续不因商品价格变化、商户状态变化或用户再次进入其他入口而改变历史订单归属。

## 4.1 商户邀请码与上下级归属

邀请码只用于确定商户注册来源和商户供货归属，不用于计算邀请奖励、拉人奖励或返佣。

规则：

1. 平台可以生成多个平台邀请码。使用平台邀请码注册并审核通过的商户为一级商户。
2. 平台后台可以不经过邀请码注册页，手工创建一级商户和店铺，用于熟人或内部渠道；手工创建必须记录创建人、初始账号交付状态和审计日志。
3. 平台后台手工创建入口只能创建一级商户，不能绕过邀请码关系创建二级或三级。
4. 每个一级商户可以生成自己的商户邀请码。使用一级商户邀请码注册并审核通过的商户为该一级商户的二级商户。
5. 如启用三级，每个二级商户可以生成自己的商户邀请码。使用二级商户邀请码注册并审核通过的商户为该二级商户的三级商户。
6. 三级商户不得生成用于注册四级商户的邀请码。
7. 一个商户只能有一个有效上级；不得同时挂靠多个一级或二级。
8. 平台可拥有多个一级商户；每个一级商户可拥有多个二级商户；每个二级商户可拥有多个三级商户。
9. 邀请码注册后形成的上下级关系必须进入审核流程，未审核或未缴保证金不得销售。
10. 平台后台可查看完整邀请/供货关系；商户后台只能查看自己可管理范围内的下级。

## 5. 服务费规则

平台服务费不是写死规则，必须由后台配置决定。

服务费配置必须支持：

1. 开启服务费。
2. 关闭服务费。
3. 调整服务费比例。
4. 查看当前生效比例。
5. 保存每笔订单创建时的服务费配置快照。

默认建议值可以是千五，即 `service_fee_bps=50`，表示 0.5%。但这只是默认配置，不是代码硬编码。后台关闭服务费时，订单服务费比例必须为 0，服务费金额必须为 0。

P0 金额口径必须区分以下字段：

```text
商品最终销售金额 final_sale_price_cents
优惠券抵扣 coupon_discount_cents
用户已确认支付 paid_amount_cents
外部支付金额 external_paid_cents
余额支付金额 balance_paid_cents
平台券补贴 platform_coupon_subsidy_cents
服务费是否开启 service_fee_enabled
服务费比例 service_fee_bps
服务费金额 service_fee_cents
服务费计算基准 service_fee_basis_amount_cents
```

平台发放的优惠券属于平台补贴，不得减少平台供货价、一级转供价、二级转供价或各级价差结算基准。服务费默认按最终销售金额计算；如果后台配置为按用户实际支付金额计算，必须写入订单快照，并重新评估商户价差影响。

建议计算口径：

```text
服务费 = 服务费计算基准金额 * 后台配置的 service_fee_bps / 10000
```

如果后台关闭服务费：

```text
服务费 = 0
service_fee_bps = 0
```

例如：

```text
商品挂牌销售金额：150.00
平台券抵扣：10.00
用户实付：140.00
平台券补贴：10.00
服务费计算基准：150.00
平台供货价：100.00
后台服务费比例：0.5%
服务费：150.00 * 0.5% = 0.75
```

关闭服务费时：

```text
服务费计算基准：150.00
后台服务费比例：0
服务费：0.00
商户预计收益：150.00 - 100.00 = 50.00
```

服务费退款口径：

1. 未履约全额退款：服务费退回，不计入平台服务费收入。
2. 平台责任导致的退款：服务费退回，由平台承担。
3. 商户责任导致的退款：服务费默认不退；如平台仲裁决定退回，计入商户责任金额，可从商户未结算应付/应收或保证金追扣。
4. 用户原因退款：服务费默认不退。
5. 部分退款：服务费按退款金额占用户实付金额的比例退回；责任归属决定由平台承担还是商户承担。
6. 使用平台券的订单退款后，买家只退实际支付金额；已使用优惠券作废，不退回买家券包，平台券补贴冲回或计入平台营销成本，具体由退款责任决定。

## 6. 收益规则

平台商品订单的商户价差收益建议计算：

```text
商户价差收益 = 服务费计算基准金额 - 平台供货价 - 平台服务费
```

三级以内商户供货订单的建议计算：

```text
一级商户价差 = 一级转供价 - 平台供货价
二级商户价差 = 二级转供价 - 一级转供价
三级销售差价 = 服务费计算基准金额 - 二级转供价 - 平台服务费承担项
```

示例：

```text
平台供货价：100.00
一级转供价：110.00
二级转供价：130.00
商品挂牌销售金额：150.00
用户实付金额：140.00
平台券补贴：10.00
一级商户价差：10.00
二级商户价差：20.00
三级销售差价：20.00，服务费按配置扣减
```

商户自有商品订单的商户收益建议计算：

```text
商户收益 = 服务费计算基准金额 - 平台服务费
```

若商户自有商品未来存在平台代履约成本，则需要单独增加成本项。

平台自营订单的建议计算：

```text
平台自营毛收益 = 服务费计算基准金额 - 履约成本 - 支付手续费/平台内部服务成本 - 平台券营销成本
```

支付手续费按支付方式记录：支付宝、微信、e支付统一按 1% 写入订单快照；余额支付手续费为 0。平台自营订单不生成商户供货应付。

所有金额必须在订单创建或支付成功时生成快照，避免后续价格调整影响历史订单。订单快照至少包含 `final_sale_price_cents`、`paid_amount_cents`、`external_paid_cents`、`balance_paid_cents`、`service_fee_enabled`、`service_fee_bps`、`service_fee_cents`、`service_fee_basis_amount_cents`、`payment_fee_bps`、`payment_fee_cents`、`payment_channel_fee_cents`、`platform_coupon_subsidy_cents`、优惠券快照和各级价差结算字段。

三级以内商户订单必须额外固化：一级商户、二级商户、三级商户、各级店铺、平台供货价、一级转供价、二级转供价、最终售价、一级价差、二级价差、三级销售价差、服务费承担规则和商户授权状态。

平台自营订单必须额外固化：`sales_channel_type=platform_self_operated`、平台自营店、最终售价、履约成本、支付手续费、平台自营毛收益和自营规则快照。这里的 `sales_channel_type=platform_self_operated` 只是订单快照和展示枚举，不是第二套平台主体；平台自营归属仍以 `shops.owner_type=platform`、`shops.merchant_id=null`、`orders.merchant_id=null` 为准。

## 7. 保证金规则

商户入驻需要缴纳保证金。保证金用于覆盖退款追扣、违规扣罚、投诉赔付、恶意刷单、虚假宣传和售后失责等风险。

建议规则：

1. 未缴纳保证金的商户不能正式开店销售。
2. 未缴纳或未被后台确认缴纳保证金的商户，不能选择平台商品上架，不能代理一级/二级上级开放的商品，不能配置转供价，不能新增销售订单。
3. 平台后台可以人工确认保证金是否缴纳，适用于熟人手工开店或线下收款场景；确认时必须记录金额、凭证、操作人、时间和备注，并同时写保证金流水、财务流水/ledger 和审计日志。
4. 保证金不足时，平台可以限制上架、暂停店铺、限制供货或冻结供货应付/服务费对账。
5. 保证金扣减必须有原因、金额、关联订单或关联事件。
6. 商户退出时，如果没有未完成订单、退款、投诉、冻结或追扣，经过观察期后退还保证金。
7. 观察期建议为 15-30 天。

保证金金额、缴纳方式和退还周期需要业务方最终确认。

## 8. 履约规则

虚拟商品履约可以是自动发放或人工发放。

自动发放适用于卡密、账号、兑换码等标准化库存商品。对有卡密的虚拟卡密商品，也就是 `code_pool` 自动发码商品，购买密码是 H5 下单必填项。买家后续提取卡密时只输入购买密码；服务端只保存购买密码哈希，不保存明文。

`code_pool` 自动发码商品下单时联系电话必填，必须校验为合法中国大陆手机号，并保存到订单快照。买家邮箱为选填项；不填邮箱不影响下单和购买密码取卡密。买家填写邮箱时，系统在确认收款并发放权益后必须触发邮件投递，把该订单已分配的卡密/权益发送到邮箱。邮件发码只是补充交付通道，不得替代购买密码提取。

`manual` 人工交付商品没有卡密池，不强制购买密码和联系电话；人工交付所需联系方式、客服微信/QQ 和交付说明按商品或店铺配置展示。

生产要真实发邮件时，必须配置邮件服务、发件人、模板、投递记录、失败状态、重试和人工补发能力。未配置邮件服务时，不能提示邮件已发送成功；只能明确显示邮件能力未配置或投递失败。

个人收款人工确认、商户支付模拟/测试回调查单确认或余额扣减成功后，系统从库存中占用并发放卡密，买家从确认成功页或订单详情进入独立提取页，仅输入购买密码即可查看卡密；订单上下文必须来自安全跳转 token、订单查询凭证或登录态订单详情，不要求买家再次输入订单号。购买密码错误 3 次锁定 30 分钟；成功提取后允许重复查看；订单退款后不得继续查看卡密，也不得邮件补发。后台必须记录发放凭证、库存状态、邮件发送、提取尝试和锁定记录。

人工发放适用于需要客服确认、账号绑定、人工核验或非标准交付的虚拟商品。买家支付后订单详情展示本店客服微信和客服二维码，引导买家添加客服领取卡密或账号；人工发放完成后由后台记录履约凭证。

履约状态建议：

1. 未履约。
2. 履约中。
3. 履约成功。
4. 履约失败。
5. 已补发。
6. 已撤销。

履约要求：

1. 每次发放必须记录发放凭证。
2. 发放失败必须支持重试或人工补发。
3. 必须避免重复发放。
4. 退款后如果权益可撤销，应记录撤销状态。
5. 用户端应能看到权益发放结果。
6. 自动发放商品不得在未支付订单、已退款订单或购买密码未校验通过时展示卡密。
7. 人工发放不得绕过订单履约记录和售后记录。

## 9. 退款规则

退款按用户实际支付金额处理，而不是只退供货价或只退商户差价。

### 9.1 未履约退款

虚拟权益未发放、未激活、未使用时，建议支持全额退款。

全额退款时：

1. 用户退回实际支付金额。
2. 平台退回供货价部分。
3. 商户退回差价部分。
4. 未履约全额退款时，服务费建议退回。

### 9.2 已履约未使用退款

如果权益已发放但可证明未使用、未下载、未激活，可以支持全额退款，并撤销权益。

必须具备权益状态判断能力，否则建议进入人工审核。

### 9.3 已使用或已激活退款

已使用、已激活、已下载或已消费的虚拟权益，默认不支持无理由退款。

但以下场景可以人工仲裁：

1. 平台履约失败。
2. 商品描述不符。
3. 商户虚假宣传。
4. 用户无法正常使用且责任不在用户。

### 9.4 责任归属

退款责任分为：

1. `platform`：平台责任，供货失败、权益不可用、平台商品描述错误。
2. `merchant`：单层商户责任，虚假宣传、私自承诺、售后失责、自有商品问题。
3. `first_tier`：多级商户链路中的一级商户责任，例如转供商品配置错误、一级承诺或一级售后失责。
4. `second_tier`：多级商户链路中的二级商户责任，例如二级转供价配置、二级承诺或二级售后失责。
5. `third_tier`：多级商户链路中的三级商户责任，例如最终销售宣传、客服承诺、三级自有商品或售后失责。
6. `user`：用户原因，误购、不想要、已使用后主观退款。
7. `mixed`：混合责任，平台、一级、二级、三级或用户中多个主体共同承担。

系统需要在售后单中记录责任归属。
单层订单使用 `merchant`；多级商户订单不得只写单一责任主体，必须明确映射到 `first_tier`、`second_tier` 或 `third_tier`，混合责任时记录各主体承担金额。

### 9.5 退款拆账矩阵

退款按用户实付金额发起。售后单必须记录退款总额、平台承担金额、商户承担金额、服务费处理方式和责任归属。

| 场景 | 用户退款 | 平台承担 | 商户承担 | 服务费处理 |
| --- | --- | --- | --- | --- |
| 未履约全额退款 | 用户实付金额 | 平台供货价部分 | 商户差价部分 | 退回 |
| 平台责任退款 | 平台仲裁金额 | 平台供货价及平台责任赔付 | 原则上不承担，除非有商户共同责任 | 退回，由平台承担 |
| 商户责任退款 | 平台仲裁金额 | 正常供货价不承担额外损失 | 商户差价及商户责任赔付 | 默认不退；若退回，由商户承担 |
| 用户原因退款 | 平台审核金额 | 按实际撤销权益情况处理 | 按实际退款差价处理 | 默认不退 |
| 混合责任退款 | 平台仲裁金额 | 平台录入承担金额 | 商户录入承担金额 | 平台在售后单中指定承担方 |

如果订单尚未完成供货应付/服务费结算，责任金额从该订单相关应付/应收中扣减；如果已经结算，进入追扣流程。

三级以内商户退款必须记录平台、一级、二级、三级、用户或混合责任。售后单需要记录用户退款金额、平台承担金额、一级承担金额、二级承担金额、三级承担金额、服务费退还金额和服务费承担方。未结算订单直接冻结并扣减供货应付/服务费账单；已结算订单按责任主体生成追扣。

三级以内商户退款矩阵：

| 场景 | 用户退款 | 平台承担 | 一级承担 | 二级承担 | 三级承担 | 服务费处理 |
| --- | --- | --- | --- | --- | --- | --- |
| 未履约全额退款 | 用户实付金额 | 平台供货价部分 | 一级商户价差 | 二级商户价差 | 三级销售差价 | 默认退回，按配置指定承担方 |
| 平台责任退款 | 平台仲裁金额 | 平台供货价及平台责任赔付 | 原则上不承担 | 原则上不承担 | 原则上不承担 | 退回，由平台承担 |
| 一级责任退款 | 平台仲裁金额 | 正常供货价不承担额外损失 | 一级商户价差及责任赔付 | 按实际责任处理 | 按实际责任处理 | 默认不退；若退回，由一级承担或按仲裁分摊 |
| 二级责任退款 | 平台仲裁金额 | 正常供货价不承担额外损失 | 原则上不承担 | 二级商户价差及责任赔付 | 按实际责任处理 | 默认不退；若退回，由二级承担 |
| 三级责任退款 | 平台仲裁金额 | 正常供货价不承担额外损失 | 原则上不承担 | 原则上不承担 | 三级销售差价及责任赔付 | 默认不退；若退回，由三级承担 |
| 混合责任退款 | 平台仲裁金额 | 平台录入承担金额 | 一级录入承担金额 | 二级录入承担金额 | 三级录入承担金额 | 平台在售后单中指定承担方 |

平台自营退款：

1. 用户退款按用户实付金额处理。
2. 平台承担平台自营订单的退款损失、履约成本处理和通道费处理。
3. 已履约权益如可撤销，必须先记录撤销或失效凭证。
4. 平台自营退款不生成商户供货链追扣。

## 10. 供货应付、服务费与追扣对账

P0 采用商户自己的收款码/收款链接收款，买家货款直接进入销售商户账户。因此本项目 P0 不做“平台代收后 T+1 给商户打款”的结算模型。

系统仍需生成供货应付、服务费、价差、退款责任和追扣账单，用于平台、上级商户和下级商户之间对账。账单可按日生成，但不是买家货款 T+1 出款。

建议结算口径：

1. 结算基准从履约成功后开始，不从下单或待确认收款开始。
2. 退款中、投诉中、风控冻结、履约失败订单不得进入可结算账单。
3. 平台服务费、平台供货应付、一级供货应付、二级供货应付、各级价差都必须按订单快照计算。
4. 商户自收款场景下，系统展示“待结算/已结算/已追扣/已冻结”，避免写成平台给商户 T+1 打款。

订单进入可结算账单前必须满足：

1. 已确认收款或真实支付成功。
2. 履约成功。
3. 未退款。
4. 未处于退款中。
5. 未被风控冻结。
6. 未被投诉冻结。

结算状态建议：

1. 待结算。
2. 冻结中。
3. 可结算。
4. 结算中。
5. 已结算。
6. 已追扣。

第一版不建设商户自助提现能力，也不建设平台代收后给商户打款能力。普通商户前台收款只能使用该商户自己绑定、审核启用的收款配置；平台收款方式只用于平台自营店，不能作为普通商户店铺兜底。平台后台负责订单对账、供货应付、服务费、支付手续费、追扣、保证金和审计。客户充值和余额支付纳入本轮；余额消费不收 1% 手续费。

三级以内商户供货结算：

1. 按订单供货链分别生成平台、一级、二级、三级相关供货应付和价差明细。
2. 同一订单同一角色不能重复结算。
3. 结算明细必须包含平台供货价、一级转供价、二级转供价、最终售价、服务费、一级差价、二级差价、三级销售差价。
4. 如线下发生上游收款或补款，后台只记录结算凭证、操作人和审计日志，不作为买家货款代付流程。
5. 追扣顺序分别适用于一级、二级和三级：未结算应付/应收 -> 待确认结算金额 -> 保证金。

平台自营订单不进入商户供货结算。平台自营订单进入平台收入、履约成本、支付手续费、退款、ledger 和对账报表。

## 11. 追扣规则

如果订单已结算后发生退款、投诉赔付或违规扣罚，平台需要从责任商户后续应付/应收、保证金或线下补款中追扣。

追扣顺序建议：

1. 优先从未结算应付/应收中扣。
2. 不足部分从待确认结算金额中扣。
3. 仍不足时从保证金中扣。
4. 保证金不足时限制店铺销售、限制供货或冻结账单。

所有追扣都必须可追溯。

## 12. 风控规则

第一版至少支持人工风控：

1. 冻结订单。
2. 解冻订单。
3. 冻结店铺。
4. 禁用店铺。
5. 限制供货、冻结账单或暂停店铺销售。
6. 商品下架。
7. 保证金扣减。

建议触发风控的情况：

1. 退款率过高。
2. 投诉率过高。
3. 商户价格异常高。
4. 商户宣传违规。
5. 履约失败过多。
6. 保证金不足。
7. 疑似刷单。

## 13. 数据隔离规则

商户只能查看自己的店铺、商品、订单、用户购买记录、收益和售后。

商户不能查看：

1. 其他商户的订单。
2. 其他商户的客户。
3. 平台全局经营数据。
4. 平台供货成本之外的内部财务数据。

平台后台按照角色权限控制数据访问。

## 14. 收款和支付状态规则

支付成功只能来自以下来源：

1. 支付宝商户、微信/腾讯商户、e支付回调验签成功。
2. 支付宝商户、微信/腾讯商户、e支付主动查单确认成功。
3. 个人支付宝或个人微信由后台人工确认真实到账。
4. 余额支付扣减成功。

付款截图、前端跳转返回、买家点击“我已付款”、二维码打开成功，都不能让订单变成已支付，也不能触发自动发码。

收款方式规则：

1. 支付宝商户、微信/腾讯商户、e支付是回调/查单自动确认。
2. 个人支付宝、个人微信是人工确认。
3. 余额支付是余额扣减自动确认。
4. 支付宝、微信、e支付、个人支付宝、个人微信统一按 1% 支付手续费写入订单和支付快照。
5. 余额支付不收 1% 手续费。
6. 平台服务费和客户支付手续费是两件事，不能混算。
7. 平台收款配置只用于平台主店；商户店只能使用该商户自己的收款配置；余额支付按平台能力单独展示。
8. 第三方支付的 `notify_url` 必须使用 API 公网地址，`return_url` 必须使用 H5 前台地址；前端返回页不能代替服务端回调或查单。

订单支付状态建议：

| 场景 | 支付状态 | 履约状态 |
| --- | --- | --- |
| 接口收款未支付 | 待支付 | 未开始 |
| 接口收款回调或查单成功 | 已支付 | 自动发码或待人工交付 |
| 接口收款金额不一致 | 支付异常 | 暂停发货 |
| 个人收款已下单 | 待确认收款 | 未开始 |
| 个人收款人工确认 | 已支付 | 自动发码或待人工交付 |
| 余额支付成功 | 已支付 | 自动发码或待人工交付 |
| 退款完成 | 已退款 | 禁止继续查看卡密 |

## 15. 优惠券规则

优惠券只做平台金额券，不做复杂活动。

1. 平台可以给所有用户一次性发放，也可以给指定用户发放。
2. 首次注册自动赠送一张注册券。
3. 优惠券一次订单最多用一张，不可叠加。
4. 优惠券可设置有效期、状态和适用商品范围。
5. 优惠券可用性、抵扣金额、核销、退款作废全部以后端为准。
6. 金额券抵扣不得影响平台供货价、一级转供价、二级转供价或各级价差结算基准。
7. 退款后已使用优惠券作废，不退回买家券包；平台券补贴按退款责任冲回或计入平台营销成本。
8. 停用、过期、未生效、已核销、已作废优惠券不得被使用。

## 16. 权限和状态机

平台后台可以按 RBAC 管理全平台数据；商户后台只能操作自己主体的数据。上级商户可以审阅下级经营汇总，但不能代下级修改店铺、商品、卡密、订单、收款、售后、结算或账号资料。

高风险动作必须写审计：

1. 登录、登出、修改密码、禁用账号、角色变更。
2. 商品创建、编辑、上架、下架、冻结、审核。
3. 商户审核、保证金确认、保证金扣减、风控冻结。
4. 收款配置保存、测试、启用、停用、查看密钥状态。
5. 人工确认收款、支付异常处理、退款审批、人工确认退款。
6. 卡密导入、查看明文、复制、导出、作废。
7. 结算单生成、确认、追扣、冲正、导出。

核心状态建议：

| 对象 | 状态 |
| --- | --- |
| 商户 | 待审核、审核拒绝、待缴保证金、可经营、冻结、禁用 |
| 店铺 | 草稿、待完善、营业中、暂停、冻结、关闭 |
| 平台商品 | 草稿、上架、下架、冻结 |
| 商户自有商品审核 | 待提交、待审核、审核通过、审核拒绝、下架、冻结 |
| 商户代理商品 | 已代理、上架、下架、冻结 |
| 收款配置 | 草稿、待测试、已启用、停用、异常 |
| 订单 | 待支付、待确认收款、已支付、履约中、已完成、退款中、已退款、异常 |
| 履约 | 未开始、锁码中、已锁码、已发码、待人工交付、已人工交付、失败、已撤销 |
| 售后 | 待处理、商户处理中、平台介入、同意退款、拒绝退款、退款中、已完成 |
| 结算 | 待结算、冻结中、可结算、结算中、已结算、已追扣 |

## 17. 卡密、购买密码和人工交付

卡密是独立业务对象，不得混入商品详情、库存备注、人工交付说明或客服资料。

1. 平台商品卡密由平台管理；商户代理平台商品不能查看平台卡密明文。
2. 商户自有商品卡密由该商户管理；上级商户审阅下级经营数据时不能查看下级自有卡密明文。
3. 自动卡密商品在订单创建或支付前置流程中必须防并发抢码，支付确认后发放；重复确认不得重复发码。
4. 卡密入库至少保存加密密文、脱敏预览和哈希，不在普通列表、导出、日志和页面源码中出现明文。
5. 查看、复制、导出卡密明文必须有独立权限和审计。
6. 买家只能查看已分配给自己订单的卡密；退款后禁止继续查看。
7. 购买密码是 `code_pool` 自动发码商品的 H5 下单必填项，只用于买家提取已发放卡密，服务端只保存哈希，不保存明文。
8. 提取页 token 不得包含卡密、购买密码、订单号明文或用户敏感信息；提取接口必须限流、锁定并禁止缓存。
9. 人工交付商品不显示卡密池入口，不导入卡密，不自动发码；商户在订单详情确认已交付并写审计。

## 18. 外部依赖和安全底线

生产运行需要明确以下外部依赖：域名和 HTTPS、PostgreSQL/Neon、对象存储、邮件服务、日志/监控、备份回滚、环境变量和密钥管理。

安全底线：

1. 数据库连接串、支付密钥、私钥、证书、上报 token 不得提交到 git，不得打印到日志，不得下发给前端。
2. 生产环境不得启用 demo 登录、mock 支付、内存数据库兜底或固定测试数据。
3. 所有金额以分为单位计算和存储，前端不得自行计算最终金额。
4. 公开接口需要限流和锁定，尤其注册、登录、邀请码、订单查询、优惠券核销、购买密码提取。
5. 可输入内容必须防 XSS、`javascript:` 链接、超长内容和导出注入。
