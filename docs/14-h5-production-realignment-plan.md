# H5 生产化重整计划

本文是 2026-05-28 后的新优先级依据。若旧文档仍提到“微信小程序优先”“前端 demo store”“固定 shop-1/agent-1 快捷操作”，以本文为准。

最新偏差审计见 `docs/15-implementation-alignment-audit.md`。其中“购买密码独立提取页、邮箱可选发码、商户自有卡密库存、无硬编码门禁”是当前 P0 防跑偏重点。

## 1. 新目标

ToSell 改为 H5 独立虚拟商品小店系统：平台后台 + 商户后台 + 多商户独立店铺前端。项目不再以微信小程序为交付目标，不再要求微信授权登录；微信只作为可选收款/支付通道之一。

P0 交付必须做到：

1. 商户拥有自己的店铺页面、商品、价格、库存、客服、收款通道、订单和售后。
2. 平台账号能管理全部商户、店铺、商品、订单、收款通道、卡密、售后、供货清算和风控。
3. 一级、二级、三级商户只能看到自己权限范围内的数据；不得在前台或后台看到其他无关商户店铺入口。
4. 后台和 H5 前台共用同一套数据库/API。后台上架、改价、库存、卡密、店铺装修、客服二维码、收款码保存后，前台立即从接口展示。
5. 删除小程序交付面，不再维护小程序页面、微信小程序登录和小程序支付入口。
6. 支付当前阶段做“商户收款通道绑定”，不是写死 mock。真实回调后续接入，但模型、后台、前台展示必须先完成。
7. H5 前台按 Amazon 风格重做：白底、深色顶部栏、橙色主按钮和价格强调、清晰分类、搜索、商品网格、详情购买区、订单查询和客服入口。
8. 自动卡密商品支持买家选填邮箱；商品后台可配置是否启用提取码。启用提取码时确认收款或后续真实支付成功后进入独立提取页，只输入购买密码/提取码即可提取；三次错误锁定 30 分钟，退款后不可查看。未启用提取码时订单详情按权限展示权益。
9. 平台优惠券支持首次注册无门槛赠券、商品范围、有效期、作废、不可叠加；平台券由平台补贴，不影响各级供货价、转供价和价差结算基准。
10. 商户使用自己的收款码/链接收款，P0 不做平台代收后 T+1 给商户打款；系统做供货应付、服务费、追扣、保证金和对账。
11. 平台后台必须支持手工创建一级商户和店铺，用于熟人或内部渠道，并生成初始账号密码交付。
12. 陌生商户必须通过商户入驻申请页输入邀请码注册，审核通过后再进入待缴保证金状态。
13. 保证金确认是硬门槛：未确认前不能销售、不能选品上架、不能代理平台或上级商品、不能配置转供价、不能创建可支付订单。

## 2. 支付与收款通道口径

支付不是固定 mock，也不是只接平台账号。每个商户可以绑定自己的收款通道。

P0 支持四类通道：

1. 支付宝个人收款码。
2. 支付宝商户收款码或跳转链接。
3. 微信个人收款码。
4. 微信商户支付码或跳转链接。

通道字段至少包括：

1. `shop_id`、`agent_id`、`owner_type`。
2. `channel_type`：`alipay_personal_qr`、`alipay_merchant_qr`、`wechat_personal_qr`、`wechat_merchant_qr`。
3. `display_name`、`account_name`、`qr_url`、`payment_url`。
4. `status`：`draft`、`pending_review`、`active`、`disabled`、`rejected`。
5. `review_status`、`reviewed_by`、`reviewed_at`、`reject_reason`。
6. `is_default`、`sort_order`、`daily_limit_cents`、`single_order_limit_cents`。
7. `created_at`、`updated_at`。

前台下单后：

1. 后端按店铺读取可用通道。
2. 用户选择支付方式。
3. 系统创建订单和支付记录，支付记录绑定通道快照。
4. 个人码阶段展示二维码和收款说明，订单进入 `pending_payment_confirmation`。
5. 商户码/跳转链接阶段可打开 `payment_url`，后续补真实回调。
6. 后台或商户确认收款后订单变为已支付，并触发自动发码或人工履约。

mock 只允许开发环境验证状态机；生产环境不能默认展示 mock，不能让用户看到 mock 支付。

## 3. 权限与店铺可见性

核心价格隔离规则：

1. 商品资料可以沿供货链开放给下级用于选品和销售。
2. 平台给一级代理的供货价只对平台和对应一级代理可见。
3. 二级代理只能看到一级给自己的一级转供价，不能看到平台给一级的供货价。
4. 三级代理只能看到二级给自己的二级转供价，不能看到一级转供价和平台供货价。
5. 价格隔离不仅是页面隐藏，API 响应、导出、浏览器网络返回、后台表格和订单列表都不能泄露越级供货价。

邀请码归属规则：

1. 平台后台可手工创建一级商户和店铺，用于熟人或内部渠道；手工创建必须记录创建人、初始账号交付状态和审计日志。
2. 后台手工创建入口只能创建一级商户，不能绕过邀请码关系创建二级或三级。
3. 平台邀请码注册的商户为一级代理。
4. 一级代理邀请码注册的商户为该一级的二级代理。
5. 二级代理邀请码注册的商户为该二级的三级代理。
6. 三级代理不得生成四级邀请码。
7. 邀请码只用于商户归属和供货关系，不产生邀请奖励、拉人奖励或返佣。

保证金状态规则：

1. 商户审核通过或后台手工创建后，状态进入待缴保证金。
2. 平台后台可以人工确认保证金，用于线下收款、熟人渠道和财务复核场景。
3. 人工确认必须记录金额、凭证、操作人、确认时间、备注，并写入保证金流水、财务流水和审计日志。
4. 未确认保证金前，商户后台可以维护基础资料和店铺装修草稿，但不能正式营业。
5. 未确认保证金前，后端必须拒绝销售、选品上架、代理平台商品、代理上级商品、配置转供价和创建可支付订单。

平台账号：

1. 可查看全部商户、店铺、商品、订单、收款通道、售后、卡密、供货清算和风控。
2. 可审核商户、商品、收款通道。
3. 可冻结店铺、订单、供货清算和收款通道。

一级商户：

1. 可管理自己的店铺前台、商品、订单、客服、收款通道。
2. 可查看自己授权范围内的下级链路。
3. 不得看到无关商户的店铺入口和订单。

二级商户：

1. 可管理自己的店铺前台、商品、订单、客服、收款通道。
2. 可查看自己授权范围内的三级链路。
3. 不得看到一级其他下级、平台其他店铺或无关商户。

三级商户：

1. 只管理自己的店铺、商品、订单、客服、收款通道和售后。
2. 不显示新增下级、下级管理、四级渠道入口。

H5 前台：

1. 通过 `/s/:shopSlug` 或 `/s/:shopId` 打开单个店铺。
2. 普通买家不能看到平台自营、代理 A/B/C 这样的测试切换入口。
3. 平台账号查看所有店铺应在后台完成，不在买家前台做公开切换。

## 4. 无硬编码红线

这是 P0 硬性要求，不是优化项。生产运行代码、API 响应、后台快捷操作、H5 前台展示、收款页和导出文件都不能依赖硬编码业务数据。

必须删除或替换：

1. `apps/miniprogram` 整体交付面。
2. API 内存 `createMemoryStore` 作为生产数据来源。
3. H5 前台固定 `shop-platform/shop-1/shop-2/shop-3` 切换。
4. 后台固定 `agent-1/shop-1/prod-1/ap-code/channel-rel-1` 操作。
5. 前端固定 mock 支付按钮和 mock 回调。
6. 后台默认客服、默认收款码、默认测试店铺。
7. 商品标题、详情、图片、规格、价格、库存、卡密硬编码。
8. 平台供货价、一级转供价、二级转供价、最低限价、优惠券金额硬编码。
9. 客服微信、客服二维码、收款二维码、支付跳转链接硬编码。
10. 前端本地常量决定订单金额、服务费、差价、清算金额。
11. 前端本地常量决定商户保证金状态、可销售状态、可代理状态或可配置转供价状态。

允许保留：

1. `*.test.ts` 测试数据。
2. `packages/database/prisma/seed.ts` 和 `packages/database/src/virtual-catalog.ts` 的开发种子数据。
3. `.env.example` 的本地示例。
4. 文档中的示例说明。

## 4.1 卡密提取与优惠券口径

卡密提取：

1. 买家邮箱为选填；填写邮箱时，确认收款发码后同步发送激活码/卡密到邮箱。邮箱发码是补充通道，不是唯一取码方式。
2. 后台按商品配置是否启用提取码；提取码主要适用于虚拟充值卡、兑换码等需要二次提取的商品。
3. 启用提取码的商品，下单时买家必须填写自定义购买密码/提取码，提取码允许纯数字。
4. 服务端只保存提取码哈希，不保存明文。
5. 自动履约订单确认收款后发放卡密，买家从确认成功页或订单详情进入提取页。
6. 提取页必须是独立页面，且只输入购买密码/提取码；订单上下文来自安全跳转 token、订单详情或订单查询凭证，不要求买家再次输入订单号。
7. 未启用提取码的自动发放商品，订单详情通过身份/订单凭证校验展示权益。
8. 提取错误 3 次锁定 30 分钟；锁定、失败、成功、退款后拒绝都写入提取日志。
9. 已支付未退款订单允许重复查看卡密；退款完成后不可查看。
10. 邮件发送必须有发送记录、失败原因、重试/人工补发入口和审计；退款后禁止再次邮件补发卡密。

优惠券：

1. 平台后台管理券模板、适用商品、有效期、发放上限、发放记录、使用记录和作废。
2. 首次注册/登录赠送无门槛券。
3. 单笔订单只能使用一张券，不可叠加。
4. 平台券是平台补贴，订单必须同时快照挂牌销售金额、券抵扣、用户实付、平台券补贴和结算基准。
5. 退款完成后已使用券作废，不退回买家券包。
6. 平台券不得降低平台供货价、一级转供价、二级转供价和各级价差结算基准。

验收：

1. 后台创建/修改商品，H5 商品列表和详情必须从 API 同步。
2. 后台创建/修改客服二维码，H5 客服入口必须从 API 同步。
3. 后台创建/修改收款通道，H5 支付页必须从 API 同步。
4. 后台导入卡密，H5 库存和支付后发码必须从数据库同步；商户自有自动卡密商品必须通过商户 scoped 库存接口管理，不能复用平台 admin 卡密接口越权操作。
5. 硬编码扫描不得在生产运行代码中发现固定测试店铺、商品、商户、收款码和 mock 支付入口。

## 5. 前台 Amazon 风格设计方向

H5 店铺要像小型 Amazon 商品页，而不是营销页。

视觉规则：

1. 顶部深色导航栏，包含店铺名、搜索框、订单查询、客服入口。
2. 主色使用 Amazon 风格橙色作为购买按钮、价格强调和选中状态。
3. 页面主体白底，灰色分隔线，卡片轻边框，不使用大面积渐变。
4. 商品列表为高密度网格：图片、标题、标签、价格、库存、销量、发货方式、购买按钮。
5. 商品详情页采用左图右购买信息，下方详情模块。
6. 移动端保留底部购买条，价格和购买按钮始终清晰。
7. 字体层级克制：标题、价格、正文、辅助说明尺寸稳定，不随视口乱缩放。

核心页面：

1. 店铺首页。
2. 分类/搜索结果。
3. 商品详情。
4. 下单页。
5. 支付通道选择/收款码页。
6. 订单查询。
7. 订单详情和卡密提取。
8. 售后申请。
9. 买家优惠券列表和下单可用券选择。

## 6. 后台与前台联动

后台必须直接驱动前台：

1. 店铺装修保存后，H5 顶部、店铺名、公告、客服二维码更新。
2. 商品创建/上架/下架/改价后，H5 商品列表和详情同步。
3. 库存和卡密导入后，H5 显示可售库存，支付后自动发码。
4. 收款通道审核启用后，H5 收款通道页显示对应二维码或跳转按钮。
5. 售后规则和商品详情更新后，H5 详情页同步。
6. 商户权限变化后，后台菜单和接口数据同步收敛。
7. 优惠券配置、适用商品、作废和发放记录保存后，H5 下单页可用券同步。
8. 提取日志、锁定状态和退款后禁看状态在后台订单详情同步。
9. 后台人工创建一级商户、确认保证金后，商户后台权限和 H5 店铺可营业状态必须从数据库/API 即时同步。
10. 保证金未确认时，后台按钮可展示但必须禁用并显示原因；后端接口也必须拒绝相关动作。

## 8. 当前最大偏差

1. 卡密提取仍偏订单详情弹窗和 `orderNo` 路径接口，尚未达到“独立提取页只输入购买密码/提取码”的最终口径。
2. 邮箱目前主要是订单字段，缺少真实邮件发送服务、发送记录、失败重试、人工补发和退款后禁发闭环。
3. 商户自有卡密商品的库存管理仍不完整；平台 admin 卡密导入不能替代商户 scoped 卡密导入、查看、作废、补发和审计。
4. 样式调整后必须回归商品详情、下单、支付通道、订单详情、提取页和售后入口，不能为了简化页面误删业务入口。
5. H5 根路径、本地种子商品和默认店铺只能作为开发初始化或便捷入口；生产运行不能依赖固定 `shop-1`、固定商品、固定图片或固定客服/收款码。
6. 优惠券、提取日志、买家用户、销售统计、邮箱发码和商户库存必须补齐后台、API、数据库和测试闭环。
