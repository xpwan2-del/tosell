# 后台收款绑定与付款截图下线计划

本文定义 ToSell 后台“收款绑定中心”的产品和技术方案，并明确 H5 前台删除“上传付款截图”主流程入口的改造目标。本文是支付配置和订单收款链路的执行基线；在开始编码前，PM、后端、前端、测试、评审必须按本文确认边界。

## 1. 目标

1. 后台支持四种收款方式：支付宝商户、腾讯/微信商户、e支付、个人支付宝。
2. 支付宝商户、腾讯/微信商户、e支付走支付创建、异步通知、主动查单和验签，系统自动更新订单支付状态。
3. 个人支付宝只作为人工收款方式，不承诺官方自动回调，不自动触发发货。
4. H5 前台删除“上传付款截图”作为核心付款入口，避免做一个将来真实支付接入后还要拆掉的流程。
5. 收款方式和发货方式是两条独立线。一个商品不需要为了不同收款方式重复上架。
6. 商品履约继续只分两种：`code_pool` 自动发码、`manual` 人工交付。
7. 支付成功后的发货规则不变：自动发码商品在订单确认已支付后发码；人工交付商品进入待商户处理。
8. 不做监听手机通知、抓账单、模拟登录、插件读取个人账单等灰色自动确认方案。

## 2. 收款方式能力矩阵

| 收款方式 | 后台是否可配置 | 是否支持官方自动确认 | 订单付款后动作 | 适合场景 |
| --- | --- | --- | --- | --- |
| 支付宝商户 | 是 | 是 | 回调/查单确认后自动改为已支付，再进入发货 | 正式生产主链路 |
| 腾讯/微信商户 | 是 | 是 | 回调/查单确认后自动改为已支付，再进入发货 | 正式生产主链路 |
| e支付 | 是 | 取决于接口能力；必须支持回调/查单才可自动确认 | 回调/查单确认后自动改为已支付，再进入发货 | 第三方或指定支付方式 |
| 个人支付宝 | 是 | 否 | 只能人工确认到账后再进入发货 | 过渡、小量、备用 |

说明：

1. “支付宝小商户收款码”如果背后是支付宝开放平台商户能力，并且能提供 AppID、公钥、私钥和异步通知能力，则按“支付宝商户”处理。
2. “腾讯商户”在后台命名给用户看可以叫“腾讯/微信商户收款”，技术实现按微信支付商户能力处理。
3. “e支付”必须拿到对方接口文档、商户号、密钥、下单接口、回调验签规则和查单接口后，才能启用为可用收款方式；如果某个 e支付通道没有可靠回调/查单，不得作为本轮可用收款方式上线，也不得伪装成人工确认自动发货。
4. “个人支付宝收款码”即使看起来可以扫码付款，也不能当作官方回调能力使用。
5. 如果商户使用个人支付宝，后台必须明确标记为“人工确认”，H5 不得提示“自动到账/自动发货”。

2. 支付宝商户
3. 腾讯/微信商户
4. e支付
5. 个人支付宝
6. 回调记录
7. 异常订单

### 3.1 收款概览

展示每个商户当前可用收款方式：

1. 是否已启用支付宝商户。
2. 是否已启用腾讯/微信商户。
3. 是否已启用 e支付。
4. 是否已启用个人支付宝。
5. 哪个方式是默认收款方式。
6. 最近一次回调时间。
7. 最近一次测试结果。
8. 异常订单数量。

页面必须用中文状态，不展示无意义数据库 ID。确实需要排障时，可在详情中折叠展示“技术信息”。

## 4. 腾讯/微信商户配置

后台字段：

| 中文字段 | 用途 |
| --- | --- |
| 微信支付商户号 | 微信支付分配的商户号 |
| 绑定 AppID | 公众号、小程序、App 或 H5 支付关联 AppID |
| 支付产品类型 | H5、Native 扫码、JSAPI 等 |
| API v3 密钥 | 用于回调解密和接口安全 |
| 商户 API 证书序列号 | 用于请求签名 |
| 商户私钥/证书 | 用于调用微信支付接口 |
| 回调地址 | 系统自动生成，例如 `https://888tech.club/api/callbacks/payments/wechat` |
| 启用状态 | 启用、停用、待测试 |

后台操作：

1. 保存配置。
2. 测试签名。
3. 发起测试订单。
4. 主动查询订单。
5. 查看最近回调。
6. 停用收款方式。

支付流程：

1. 买家在 H5 下单。
2. 后端生成 ToSell 订单。
3. 后端向微信支付创建支付单。
4. H5 展示微信支付二维码或跳转参数。
5. 买家支付。
6. 微信支付向 ToSell 回调地址发送支付结果通知。
7. ToSell 验签、解密、校验订单号、金额、商户号和支付状态。
8. 校验通过后，订单改为已支付。
9. `code_pool` 商品自动发码，`manual` 商品进入待人工交付。

## 5. 支付宝商户配置

后台字段：

| 中文字段 | 用途 |
| --- | --- |
| 支付宝应用 AppID | 支付宝开放平台应用 ID |
| 支付产品类型 | 当面付扫码、手机网站支付、电脑网站支付等 |
| 应用私钥 | ToSell 调支付宝接口时签名 |
| 支付宝公钥 | ToSell 验证支付宝通知签名 |
| 签名方式 | 默认 RSA2 |
| 回调地址 | 系统自动生成，例如 `https://888tech.club/api/callbacks/payments/alipay` |
| 页面返回地址 | 支付后前端跳回订单页，只做展示，不作为支付成功依据 |
| 启用状态 | 启用、停用、待测试 |

后台操作：

1. 保存配置。
2. 测试签名。
3. 发起测试订单。
4. 主动查询交易。
5. 查看最近异步通知。
6. 停用收款方式。

支付流程：

1. 买家在 H5 下单。
2. 后端生成 ToSell 订单。
3. 后端向支付宝创建支付单。
4. H5 展示支付宝二维码或支付跳转。
5. 买家支付。
6. 支付宝向 ToSell 回调地址发送异步通知。
7. ToSell 验签、校验订单号、金额、卖家/应用信息和交易状态。
8. 校验通过后，订单改为已支付。
9. `code_pool` 商品自动发码，`manual` 商品进入待人工交付。

## 6. e支付配置

e支付作为独立支付方式接入，后台必须做成可配置适配器，不能写死成某一家文案。如果用户后续指定具体 e支付服务商，以该服务商官方接口文档为准。

后台字段：

| 中文字段 | 用途 |
| --- | --- |
| e支付服务商名称 | 标识具体 e支付通道 |
| 商户号 | e支付服务商分配的商户号 |
| 应用 ID/渠道 ID | 如果服务商要求则填写 |
| 支付产品类型 | 扫码、H5、聚合支付等 |
| 接口网关地址 | 下单、查单等接口地址 |
| 接口密钥/私钥 | 请求签名和回调验签 |
| 回调验签方式 | RSA、HMAC、MD5 等，以服务商文档为准 |
| 回调地址 | 系统自动生成，例如 `https://888tech.club/api/callbacks/payments/epay` |
| 页面返回地址 | 支付后前端跳回订单页，只做展示，不作为支付成功依据 |
| 启用状态 | 启用、停用、待测试 |

后台操作：

1. 保存配置。
2. 测试签名。
3. 发起测试订单。
4. 主动查询订单。
5. 查看最近回调。
6. 停用收款方式。

接入规则：

1. e支付必须支持服务端创建支付单。
2. e支付必须支持服务端异步通知或主动查单，至少要有一种可靠确认方式。
3. e支付必须有明确验签规则。
4. e支付回调和查单都必须校验订单号、金额、商户号、交易状态。
5. 如果某个 e支付通道不能满足以上条件，不得启用为本轮正式收款方式，不能自动确认、不能自动发码；是否另行作为人工收款方式需单独立项确认。

支付流程：

1. 买家在 H5 下单。
2. 后端生成 ToSell 订单。
3. 后端向 e支付创建支付单。
4. H5 展示 e支付二维码或支付跳转。
5. 买家支付。
6. e支付向 ToSell 回调地址发送支付结果通知，或 ToSell 主动查单。
7. ToSell 验签、校验订单号、金额、商户号和交易状态。
8. 校验通过后，订单改为已支付。
9. `code_pool` 商品自动发码，`manual` 商品进入待人工交付。

## 7. 个人支付宝收款配置

后台字段：

| 中文字段 | 用途 |
| --- | --- |
| 收款名称 | 给运营人员看的名称 |
| 支付宝账号 | 展示或备注用 |
| 收款二维码 | 展示给买家扫码 |
| 收款说明 | 告诉买家如何备注、多久确认 |
| 启用状态 | 启用、停用 |

限制：

1. 不支持官方自动回调。
2. 不自动确认订单支付。
3. 不自动发码。
4. 不允许前端文案暗示“付款后自动到账”。
5. 只能进入“待确认收款”订单队列，由商户或平台按权限确认。

个人收款可作为过渡能力，但不能作为正式自动支付主链路。

## 8. H5 前台删除上传付款截图主流程

必须删除或降级的内容：

1. 下单支付页不再把“上传付款截图”作为付款步骤。
2. 订单详情页不再把“上传付款截图”作为主要操作按钮。
3. 付款成功判断不再依赖客户上传截图。
4. 自动发码不再由截图审核触发。

保留的可选能力：

1. 如果订单使用个人收款，订单详情可以保留“补充付款说明/联系客服”的入口。
2. 付款截图最多作为争议材料或过渡辅助证据，不进入主流程。
3. 后台可以在异常订单中查看客户补充材料，但不能把它设计成正式支付确认机制。

前台付款页应改成：

1. 商家收款方式：展示“去支付/扫码支付”，等待官方支付结果。
2. 个人收款方式：展示二维码和说明，订单状态为“待确认收款”。
3. 支付完成后：展示订单状态、发货状态和取卡密入口。

## 9. 订单状态规则

| 场景 | 支付状态 | 履约状态 |
| --- | --- | --- |
| 接口收款未支付 | 待支付 | 未开始 |
| 接口收款回调成功 | 已支付 | 自动发码或待人工交付 |
| 接口收款回调失败但查单成功 | 已支付 | 自动发码或待人工交付 |
| 接口收款金额不一致 | 支付异常 | 暂停发货 |
| 个人收款已下单 | 待确认收款 | 未开始 |
| 个人收款人工确认 | 已支付 | 自动发码或待人工交付 |
| 退款完成 | 已退款 | 禁止继续查看卡密 |

支付成功只能由以下来源产生：

1. 支付宝商户、腾讯/微信商户、e支付回调验签成功。
2. 支付宝商户、腾讯/微信商户、e支付主动查单确认成功。
3. 个人收款人工确认到账。

## 10. 收款配置基础字段

每条收款配置都必须包含：

1. 配置归属：平台配置或商户配置。
2. 所属商户/店铺：商户收款配置必须绑定具体商户和店铺。
3. 收款类型：支付宝商户、腾讯/微信商户、e支付、个人支付宝。
4. 确认模式：接口回调/查单、人工确认。
5. 环境：测试环境、生产环境。
6. 启用状态：停用、待测试、启用、暂停。
7. 默认方式：同一店铺同一时间只能有一个默认收款方式。
8. 密钥状态：只显示“已配置/未配置/已过期”，不回显明文。
9. 回调地址：系统自动生成。
10. 页面返回地址：只用于前端展示，不作为支付成功依据。
11. 最近测试时间、最近测试结果、最近回调时间。
12. 创建人、修改人、启停人和审计记录。

## 11. 订单支付快照

订单创建时必须保存本次支付使用的收款方式快照，后续商户修改收款配置不能影响历史订单。

支付快照必须包含：

1. 收款方式类型。
2. 收款配置 ID。
3. 商户号、AppID、服务商标识的脱敏值。
4. 应付金额和币种。
5. ToSell 订单号。
6. 第三方支付单号和第三方交易号。
7. 支付状态。
8. 支付确认来源：回调、主动查单、人工确认。
9. 支付过期时间。
10. 支付成功时间。
11. 回调处理时间。
12. 异常原因：验签失败、金额不一致、商户不匹配、订单不存在、重复回调、已退款后回调。

## 12. 回调记录和异常订单

后台必须提供“回调记录”：

1. 来源：支付宝商户、腾讯/微信商户、e支付。
2. 订单号。
3. 第三方交易号。
4. 通知时间。
5. 验签结果。
6. 金额校验结果。
7. 幂等处理结果。
8. 错误原因。

后台必须提供“异常订单”：

1. 回调验签失败。
2. 金额不一致。
3. 商户号/AppID 不匹配。
4. 重复回调。
5. 订单不存在。
6. 已退款后又收到支付通知。
7. 已发码订单发生退款争议。

异常订单只能由有权限的后台账号处理，并写审计。

异常订单处理动作：

1. 主动查单。
2. 标记已处理。
3. 保持异常并暂停发货。
4. 添加人工备注。
5. 查看脱敏回调记录。
6. 写入审计。

异常订单在处理完成前不得自动发码，不得进入已支付履约链路。

## 13. 权限原则

1. 平台账户可以配置、查看和审计全平台收款方式。
2. 商户只能配置和查看自己商户范围内的收款方式。
3. 上级商户可以审阅下级经营汇总，但不能修改下级收款配置。
4. 商户不能查看其它商户的密钥、私钥、证书、回调原文敏感字段。
5. 密钥和证书不允许明文回显；保存后只能显示脱敏状态，例如“已配置”。
6. 查看、修改、启用、停用、测试收款配置都必须写审计。

## 14. 安全要求

1. API 密钥、私钥、证书不得提交到 git。
2. 密钥、私钥、证书必须加密存储或使用生产环境密钥管理。
3. 回调必须验签，不能只看前端跳转结果。
4. 回调必须校验订单号、金额、商户号/AppID、交易状态。
5. 回调处理必须幂等，重复通知不能重复发码。
6. 回调原始 payload 可以保存用于排障，但敏感字段必须脱敏或加密。
7. 生产环境不得启用 mock 支付作为真实付款入口。
8. 个人收款不能使用监听通知、抓账单、模拟登录等方式伪装自动回调。

## 17. 测试和验收

必须覆盖：

1. 平台配置腾讯/微信商户，保存后不回显密钥明文。
2. 平台配置支付宝商户，保存后不回显私钥明文。
3. 平台配置 e支付，保存后不回显接口密钥明文。
4. 商户配置自己的个人支付宝收款码，H5 展示二维码，但订单不自动支付。
5. H5 接口收款下单后，不再出现上传付款截图主按钮。
6. H5 个人支付宝下单后，展示二维码和“等待商户确认收款”。
7. 模拟回调验签成功后，订单变为已支付。
8. 主动查单确认成功后，订单变为已支付。
9. 重复回调不会重复发码。
10. 金额不一致进入异常订单，不发码。
11. 前端支付返回页不等于支付成功，必须等回调或查单结果。
12. `code_pool` 商品支付成功后自动发码。
13. `manual` 商品支付成功后进入待人工交付。
14. 退款完成后禁止继续查看卡密。
15. M1/M2/M3 只能配置和查看自己的收款方式，不能修改下级收款配置。
16. 平台账号可查看全局回调记录和异常订单。
17. 所有配置、测试、回调处理、人工确认都写审计。

## 18. 不做项

1. 不做手机通知监听。
2. 不做个人支付宝账单抓取。
3. 不做微信/支付宝模拟登录。
4. 不做浏览器插件读取收款记录。
5. 不把上传付款截图作为正式支付确认机制。
6. 不做在线客服、坐席、工单、SLA。
