# 功能卡片与开发计划

本文是 P0 开发排期依据。当前项目目标是 H5 独立虚拟商品小店系统，包含平台后台、商户后台、多商户独立店铺前端、数据库、API、收款通道、卡密履约、售后、供货清算和权限隔离。微信小程序不再作为 P0 交付目标。

## 1. 里程碑总览

| 阶段 | 名称 | 目标 | 负责人 |
| --- | --- | --- | --- |
| P0-0 | 文档与口径清理 | 废止小程序优先、固定 demo、mock 支付生产化等旧口径 | main + pm_architect |
| P0-1 | 数据库生产化 | 商户、店铺、商品、库存、卡密、订单、收款通道、渠道关系全部落库 | database_expert |
| P0-2 | 后端 API 与权限 | 平台/商户/买家 API、逐级价格隔离、邀请码层级、收款通道、订单履约 | backend_worker |
| P0-3 | 后台管理 | 平台后台和商户后台可完成商品、库存、卡密、订单、收款、售后、供货清算管理 | frontend_dev + backend_worker |
| P0-4 | H5 店铺前台 | Amazon 风格单店铺前台，商品列表、详情、下单、收款码、订单查询、卡密提取 | frontend_dev |
| P0-5 | 测试与上线门槛 | 全链路测试、越权测试、硬编码扫描、视觉验收、部署准备 | integration_tester + reviewer |

## 2. P0-0 文档与口径清理

目标：确保所有员工以统一开发依据工作，不被旧文档误导。

任务：

1. README 和 docs 明确 H5 为 P0，微信/支付宝只是收款或支付通道。
2. 小程序相关文档标注为历史参考，不进入 P0 开发依据。
3. 明确本项目是 B2B2C 供货价差，不是返佣、返利、团队奖或邀请奖励。
4. 明确商品资料可以向下级开放，隐藏的是越级供货价格。
5. 明确后台、API、导出、浏览器网络响应都必须执行逐级价格隔离。

验收：

1. `rg` 扫描旧文档时，小程序只出现在“历史/不作为 P0”说明中。
2. 文档不再要求小程序提审、微信授权登录或 `wx.requestPayment` 作为 P0。
3. 文档明确：二级能看到一级给自己的转供价，但看不到平台给一级的供货价。

## 3. P0-1 数据库生产化

### 卡 1.1 商户、账号与店铺

功能：

1. 平台账号、商户账号、买家 H5 身份分离。
2. 商户入驻申请、审核、保证金状态、风险状态落库。
3. 平台后台可手工创建一级商户和店铺，生成初始账号密码并记录交付状态。
4. 每个商户有独立店铺、slug/domain、店铺装修、公告、客服二维码。
5. 平台自营店使用 `owner_type=platform`，不属于一级代理。

验收：

1. 平台可查看全部商户和店铺。
2. 商户只能查看自己的店铺和授权下级范围。
3. 买家前台只能打开当前店铺，不出现公开店铺切换。
4. 后台手工创建的一级商户处于待保证金确认状态，未确认前不能销售、选品或代理商品。

### 卡 1.2 邀请码与渠道关系

功能：

1. 平台邀请码注册一级商户。
2. 一级邀请码注册该一级下的二级商户。
3. 二级邀请码注册该二级下的三级商户。
4. 三级不得生成四级邀请码。
5. 邀请码只用于商户归属和供货关系。
6. 不熟悉的商户必须通过入驻申请页输入邀请码注册；熟人一级商户可由平台后台手工创建。

验收：

1. `merchant_invite_codes.code` 唯一。
2. 一个商户同一时间只能有一个有效上级。
3. 后端拒绝四级关系创建。
4. 不产生邀请奖励、拉人奖励或返佣字段。
5. 手工创建入口只能创建一级商户，不能绕过二级/三级归属规则。

### 卡 1.3 商品、库存与卡密

功能：

1. 平台商品库：标题、图片、详情、规格、供货价、最低限价、建议售价、履约方式、售后规则。
2. 商户商品：选品上架、自有商品提交审核、售价、库存展示、上下架状态。
3. 虚拟卡密码池：批量导入、库存扣减、自动发码、手工交付记录。
4. 商品详情和图片全部来自数据库或对象存储，不允许硬编码。
5. 商户保证金未确认时，不能选品上架、不能代理平台商品或上级商户商品、不能配置转供价。

验收：

1. 后台新增/改价/下架商品后，H5 列表和详情同步。
2. 自动卡密库存不足时不可继续自动销售或进入待人工交付。
3. 历史订单保存商品快照，不受后续改价影响。

### 卡 1.4 商户收款通道

功能：

1. 支持支付宝个人收款码、支付宝商户码/链接、微信个人收款码、微信商户码/链接。
2. 商户提交收款通道，平台审核后启用。
3. 店铺可设置默认通道和排序。
4. 订单支付记录保存通道快照。

验收：

1. H5 下单后只显示当前店铺已启用收款通道。
2. 后台修改收款码后，前台支付页同步。
3. 生产环境不得展示固定 mock 支付。

## 4. P0-2 后端 API 与权限

### 卡 2.1 逐级价格隔离

核心规则：

1. 商品资料可沿供货链开放给下级选品和销售。
2. 平台给一级的供货价只对平台和对应一级可见。
3. 二级只看到一级给自己的一级转供价，不看到平台供货价。
4. 三级只看到二级给自己的二级转供价，不看到一级转供价和平台供货价。
5. 隔离必须在服务端响应层、导出层、后台表格和订单列表执行。

验收：

1. 二级商户调用平台商品/选品 API 不包含 `platform_supply_price_cents`。
2. 三级商户 API 不包含 `platform_supply_price_cents` 和 `first_tier_supply_price_cents`。
3. 导出文件按角色脱敏，不靠前端隐藏字段。

### 卡 2.2 选品与转供价

功能：

1. 一级可看到平台开放商品和平台给自己的供货价。
2. 一级选择商品上架到自己的店铺，也可开放给二级并设置一级转供价。
3. 二级只看到一级开放给自己的商品和一级转供价，可上架到自己的店铺。
4. 二级可开放给三级并设置二级转供价。
5. 三级只看到二级开放给自己的商品和二级转供价，可上架销售。

验收：

1. 低于上游供货价或最低限价的转供价/售价保存失败。
2. 转供价调整写审计。
3. 历史订单按下单时快照清算。

### 卡 2.3 订单、支付状态与履约

功能：

1. 买家创建订单时后端计算价格、优惠、服务费、库存和可售状态。
2. 买家创建订单时后端必须校验店铺和销售商户保证金确认状态；未确认保证金不得创建可支付订单，不能只靠前端禁用按钮。
3. 个人码/商户码阶段订单进入 `pending_payment_confirmation`。
4. 后台或商户确认收款后订单变为已支付并自动触发履约。
5. 自动卡密：支付成功后生成提取页记录。
6. 手工交付：支付成功后提示联系店铺客服。

验收：

1. 支付成功不需要人工确认订单本身，只需要系统检测或后台确认收款结果。
2. 自动卡密订单可在订单详情和提取页查看卡密。
3. 手工订单不显示卡密，只显示客服入口和交付状态。
4. 未确认保证金的店铺创建订单时，后端返回明确拒绝原因，不生成可支付订单或支付记录。

### 卡 2.4 卡密提取码

功能：

1. 后台按商品配置是否启用提取码；提取码主要适用于虚拟充值卡、兑换码等需要二次提取的商品，不是所有自动卡密商品都必须有提取码。
2. 买家邮箱为选填项；填写邮箱时，确认收款并发码后同步发送激活码/卡密到邮箱。
3. 启用提取码的商品，买家下单时必须自定义提取码，可为纯数字。
4. 确认收款或后续真实支付成功后，从确认成功页或订单详情进入卡密提取页。
5. 提取页只需要输入提取码即可提取，订单上下文来自安全跳转 token 或订单查询凭证，不要求买家再次输入订单号。
6. 未启用提取码的自动卡密商品，买家通过订单详情和身份/订单凭证校验查看卡密。
7. 提取尝试写入日志，记录成功/失败、IP、UA、锁定原因和操作时间。
8. 错误 3 次锁定 30 分钟。
9. 允许重复查看；退款完成后不允许再看卡密。

验收：

1. 启用提取码的商品，提取码不明文存储。
2. 未启用提取码的商品，下单页不强制填写提取码。
3. 退款后提取接口返回不可查看状态。
4. 后台可查看订单提取日志、当前锁定状态和退款后禁看状态。
5. 邮箱未填写不影响下单；邮箱已填写时发码邮件有发送记录和失败重试/人工补发入口。

### 卡 2.5 优惠券

功能：

1. 平台创建无门槛注册赠送券。
2. 平台后台支持券模板、适用商品、有效期、库存/发放上限、发放记录、使用记录和作废。
3. 优惠券可限制适用商品、店铺、有效期和状态。
4. 优惠券不可叠加使用。
5. 平台可作废优惠券。
6. 平台券由平台补贴，不影响供货价、转供价和各级价差结算基准。
7. 退款后已使用优惠券作废，不退回买家券包。

验收：

1. 新用户首次注册登录自动获得配置的赠送券。
2. 下单只能使用一张符合条件的券。
3. 过期或作废券不可使用。
4. 商品范围不匹配的券不可使用。
5. 使用平台券的订单仍按挂牌销售金额计算各级价差。
6. 退款后券状态为 `voided_after_refund`，不可再次使用。

## 5. P0-3 后台管理

### 卡 3.1 平台后台

功能：

1. 商户审核、保证金、店铺、商品、库存、卡密、收款通道审核。
2. 订单、履约、售后、退款、供货清算、追扣、风控、审计。
3. 平台自营店和平台自营商品管理。
4. 渠道关系、邀请码、转供价、价格隔离检查。
5. 优惠券模板、发放、使用、作废和商品范围管理。
6. 买家用户、订单查询凭证、提取日志、锁定状态和销售统计。

验收：

1. 平台后台没有固定 `shop-1/agent-1/prod-1` 快捷操作。
2. 每个功能都对应数据库和 API。
3. 敏感操作写审计日志。

### 卡 3.2 商户后台

功能：

1. 店铺装修、客服二维码、收款通道、商品上架、库存卡密、订单、售后、供货清算。
2. 一级商户可管理自己开放给二级的商品和一级转供价。
3. 二级商户可管理自己开放给三级的商品和二级转供价。
4. 三级商户不显示新增下级入口。

验收：

1. 商户后台只能显示自己和授权下级范围。
2. 二级后台不显示平台给一级的供货价。
3. 三级后台不显示一级转供价和平台供货价。

## 6. P0-4 H5 店铺前台

功能：

1. 店铺首页：深色顶部栏、搜索、分类、商品网格、客服入口、订单查询。
2. 商品详情：图片、标题、价格、库存、发货方式、详情说明、售后规则、购买区。
3. 下单页：选填邮箱、优惠券、支付方式、订单金额；商品启用提取码时展示必填提取码。
4. 收款码页：展示当前店铺启用通道二维码或跳转链接。
5. 订单查询：手机号/邮箱/订单号或登录态查询。
6. 卡密提取页：仅启用提取码的商品进入，输入提取码提取，三次锁定半小时。

验收：

1. 桌面和手机页面无重叠，按钮和价格清晰。
2. 页面风格参考 Amazon：白底、深色顶部栏、橙色价格和 CTA。
3. 前台不展示供货价、转供价、差价、服务费、清算和风控信息。

## 7. P0-5 测试与上线门槛

测试必须覆盖：

1. 平台创建商品 -> 一级选品 -> 一级上架 -> H5 购买。
2. 平台商品 -> 一级设置一级转供价 -> 二级选品上架 -> H5 购买。
3. 二级设置二级转供价 -> 三级选品上架 -> H5 购买。
4. 二级/三级 API、后台表格和导出不泄露越级供货价。
5. 商户绑定收款通道 -> 平台审核 -> H5 展示 -> 确认收款 -> 自动/手工履约。
6. 自动卡密、手工交付、退款后禁止查看卡密。
7. 注册赠券、商品券、有效期、作废、不可叠加。
8. 硬编码扫描：生产运行代码不得依赖固定测试店铺、商品、代理或收款码。

上线前必须通过：

1. `npm test`
2. `npm run typecheck`
3. `npm run db:validate`
4. 后台到 H5 的浏览器端到端验证。
5. 权限越权和价格泄露专项测试。
