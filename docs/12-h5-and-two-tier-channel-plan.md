# H5、平台自营与受控两级渠道供货开发计划

本文记录本次新增需求，作为后续开发、测试和 CCB 团队协作依据。该需求是在 V1/V2 单层微商小店基础上的业务模型升级，目标是支持微信小程序和 H5 独立店铺共用同一套后端，并支持平台自营店与受控两级渠道供货并存。

## 1. 产品边界

1. 微信小程序不是 HTML 页面，不能脱离微信在普通浏览器中直接打开。
2. 项目需要新增 H5 买家/店铺端，让用户可以在微信外浏览器、短信链接、普通网页链接中访问代理店铺。
3. 小程序端、H5 端、代理/渠道端和平台后台共用同一套 API、数据库、商品、订单、履约、售后、结算和风控。
4. 支付账号未开通前，只做支付渠道规划、模型占位和 mock 支付闭环，不接真实支付。
5. 支付渠道需要预留微信小程序支付、微信内 H5 JSAPI、微信外 H5 支付、支付宝手机网站支付。
6. 平台可以开设平台自营店，直接面向消费者销售。
7. 平台自营店不属于一级渠道商，不参与一级/二级供货链，不生成渠道差价结算。
8. 渠道供货只允许两级：平台供货 -> 一级渠道商 -> 二级渠道商 -> 消费者。
9. 一级渠道商可以向平台申请开通二级渠道能力，必须经过平台审核授权。
10. 二级渠道商必须独立入驻、审核、缴纳保证金，并拥有自己的店铺、客服、订单、售后和结算。
11. 二级渠道商不得继续创建三级渠道商。

## 2. 禁止边界

系统仍然不得设计为多级分销或返佣系统。禁止以下能力：

1. 三级渠道或更多层级。
2. 下级返佣、推广返佣、按比例返利。
3. 团队奖、团队业绩、平级奖、领导奖。
4. 邀请奖励、拉人奖励、招募人数奖励。
5. 代理等级、团队等级、按等级差异返利。
6. 以招募人数、团队订单、团队 GMV 作为收益依据。
7. 代理自助提现，除非后续单独确认。

推荐使用术语：

1. 一级渠道商。
2. 二级渠道商。
3. 渠道授权。
4. 平台供货价。
5. 一级转供价。
6. 最终销售价。
7. 供货价差。
8. 销售差价。
9. 供货链快照。
10. 渠道结算。
11. 平台自营店。
12. 直营店。

避免使用字段或文案：`commission`、`rebate`、`team`、`level`、`invite_reward`、`downline_commission`、`parent_agent`、返佣、团队奖、邀请奖励、代理等级。

术语统一：

1. 单层销售场景可以称为代理或渠道商。
2. 两级供货场景必须明确称为一级渠道商、二级渠道商。
3. 数据库、API 和后台配置优先使用 `channel`、`tier`、`resell_supply_price`、`channel_authorization` 等供货语义。
4. 页面文案必须强调供货价差、销售差价和渠道授权，不使用返佣、团队收益或代理等级。

## 3. 核心业务例子

### 3.1 平台自营

```text
平台自营售价：150.00
履约成本：100.00
支付通道费/内部服务成本：按配置计算，账号未开通阶段可为 0
平台自营毛收益 = 用户实付金额 - 履约成本 - 支付通道费/内部服务成本
```

平台自营订单归属为 `platform_self_operated`，不产生一级/二级渠道差价，不进入渠道打款结算，但必须进入订单、履约、退款、ledger 和对账。

### 3.2 两级渠道供货

```text
平台供货价：100.00
一级转供价：110.00
二级最终售价：150.00
一级渠道差价：110.00 - 100.00 = 10.00
二级销售差价：150.00 - 110.00 = 40.00
平台服务费：按用户实付金额和配置费率计算
```

该收益来自真实供货价格差，不是按比例返佣。订单必须固化平台供货价、一级转供价、最终售价、服务费、一级渠道差价、二级销售差价和责任归属快照。

## 4. 功能模块

### 4.1 H5 买家/店铺端

1. H5 店铺首页。
2. H5 商品列表和详情。
3. H5 下单页。
4. H5 支付占位页。
5. H5 订单详情。
6. H5 权益查看。
7. H5 售后申请。
8. H5 客服微信或联系方式展示。
9. H5 分享链接与二维码。

H5 端不得展示平台供货价、一级转供价、服务费、渠道差价、结算金额、冻结金额和风控细节。

### 4.1.1 平台自营店

1. 平台自营店首页。
2. 平台自营商品上架/下架。
3. 平台自营售价设置。
4. 平台自营订单、履约、售后、退款。
5. 平台自营经营看板。
6. 平台自营收入、成本、支付通道费和对账。

平台自营店与渠道店共用交易基础设施，但订单归属、收益口径和结算对象必须单独区分。

### 4.2 多端统一架构

1. 小程序端和 H5 端共享店铺、商品、订单、履约、售后数据。
2. 用户身份模型需要支持小程序 `openid/unionid` 和 H5 手机号/网页登录身份。
3. 商品、价格、订单、支付、退款、履约、结算、ledger 使用统一后端模型。
4. 图片、资质、售后凭证、打款凭证统一进入对象存储。
5. API 域名、H5 域名、小程序合法域名必须统一规划 HTTPS 和备案。

### 4.3 两级渠道供货

1. 一级渠道商入驻、审核、保证金。
2. 一级渠道商申请开通二级渠道能力。
3. 平台审核并授权或关闭一级的二级渠道能力。
4. 二级渠道商入驻、审核、保证金。
5. 一级给二级配置可供商品、一级转供价、上下架权限。
6. 二级配置最终销售价，后端校验不得低于一级转供价和平台最低限制价。
7. 二级拥有独立店铺、客服、订单、收益、结算和售后。
8. 系统强制二级不得继续创建三级渠道。

### 4.4 供货链价格体系

1. 平台供货价：平台给一级渠道商的供货价。
2. 一级转供价：一级给二级渠道商的供货价。
3. 最终销售价：消费者看到并支付的价格。
4. 平台服务费：默认仍为用户实付金额的 0.5%，后续可配置。
5. 一级渠道差价：一级转供价减平台供货价。
6. 二级销售差价：最终销售价减一级转供价，并按服务费承担规则扣减。
7. 订单创建或支付成功时必须生成供货链快照，历史订单不受后续调价影响。

### 4.5 后台管理

1. 一级渠道审核。
2. 二级渠道审核。
3. 一级开通二级能力审核。
4. 渠道关系管理。
5. 转供价管理与调整审计。
6. 二级店铺管理。
7. H5 店铺配置。
8. 多渠道支付配置占位。
9. 供货链订单查询。
10. 一级/二级渠道结算单。
11. 退款责任分摊与追扣。
12. 风控冻结：一级、二级、店铺、商品、订单、结算。

### 4.6 支付渠道占位

1. 统一支付渠道字段：`wechat_miniprogram`、`wechat_h5_jsapi`、`wechat_h5`、`alipay_wap`、`mock`。
2. 微信小程序内使用微信小程序支付，前端调用 `wx.requestPayment`。
3. 微信内 H5 使用微信 JSAPI 支付。
4. 微信外手机浏览器 H5 使用微信 H5 支付。
5. 普通手机浏览器支持支付宝手机网站支付 `alipay_wap`。
6. 微信小程序内不设计支付宝支付入口。
7. 支付单、支付回调、退款单、退款回调统一模型。
8. mock 支付只用于账号未开通前的本地闭环，不得绕过订单、ledger、退款、结算逻辑。
9. 真实支付接入前，H5 页面可以展示“模拟支付/渠道待开通”，后端状态机按真实支付设计。

## 5. 退款与结算规则

退款按用户实付金额处理。售后单必须记录责任归属：平台、一级、二级、用户、混合责任。

退款拆账必须记录：

1. 用户退款金额。
2. 平台承担金额。
3. 一级承担金额。
4. 二级承担金额。
5. 服务费退还金额。
6. 服务费承担方。

结算规则：

1. T+1 默认仍从履约成功时间起算。
2. 退款中、投诉中、风控冻结、履约失败订单不得结算。
3. 一级和二级分别生成结算明细。
4. 单层订单同一订单不能重复进入结算；两级渠道订单按 `order_id + settlement_role` 唯一，同一订单可以分别生成一级和二级结算明细，但同一角色不得重复结算。
5. 结算明细必须包含平台供货价、一级转供价、最终售价、服务费、一级差价、二级差价。
6. 手工打款仍为当前口径：生成结算单、导出、人工打款、回填凭证。
7. 追扣顺序分别适用于一级和二级：未结算收益 -> 待打款收益 -> 保证金。

## 6. 部署与服务器规划

建议使用国内云服务器，优先腾讯云或阿里云。正式环境至少包含：

1. API 服务：`api.example.com`。
2. H5 店铺：`shop.example.com`。
3. 平台后台：`admin.example.com`。
4. PostgreSQL 数据库。
5. Redis 缓存、限流、任务锁。
6. 对象存储：商品图、资质、售后凭证、打款凭证。
7. HTTPS 证书。
8. ICP 备案。
9. 小程序合法服务器域名配置。
10. 日志、备份、监控和告警。

## 7. 验收标准

1. H5 店铺能在微信外浏览器独立打开。
2. 小程序店铺和 H5 店铺使用同一后端商品、订单、履约和售后。
3. 平台自营店可在小程序和 H5 完成下单、履约、退款和对账。
4. 平台自营订单归属为 `platform_self_operated`，不生成渠道结算。
5. 一级渠道商可申请开通二级渠道能力，未经授权不可创建二级。
6. 二级渠道商必须审核通过并缴纳保证金后才能销售。
7. 一级可给二级设置转供价。
8. 二级可设置最终销售价并完成下单。
9. 示例订单可复算：平台 100，一级转供 110，二级售价 150，一级差价 10，二级差价 40，再按规则处理服务费。
10. 二级不可再创建三级渠道。
11. 退款、追扣、结算可按供货链拆分。
12. 审计日志覆盖平台自营配置、转供价调整、渠道授权、退款裁决、结算、打款、保证金扣减。
13. 代码、页面、API 和导出不出现返佣、团队奖、邀请奖励、代理等级、三级渠道等设计。

## 8. 待确认事项

1. 一级开通二级渠道能力的审核条件。
2. 二级渠道商保证金金额。
3. 服务费由二级承担、一级分摊，还是按系统配置决定。
4. H5 登录优先使用手机号、短信验证码，还是微信网页授权。
5. H5 域名、API 域名和备案主体。
6. 支付渠道优先级：微信小程序支付、微信 H5、支付宝手机网站支付的接入顺序。
7. 一级转供价是否允许低于平台最低销售价，但不得低于平台供货价。
8. 二级最终售价是否需要最高价或价格异常提醒。
