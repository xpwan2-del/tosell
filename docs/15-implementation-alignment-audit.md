# 实现偏差审计与补齐清单

本文记录 2026-05-28 对现有文档和实现的核对结果。后续 PM、开发、测试和 reviewer 必须以本文作为“防跑偏”检查清单之一；支付绑定和收款确认如有冲突，以 `docs/18-payment-binding-and-voucher-removal-plan.md` 为准；H5 生产口径以 `docs/14-h5-production-realignment-plan.md` 为准。

## 1. 本次确认的业务口径

1. 项目是 H5 独立虚拟商品小店，不再以微信小程序为 P0。
2. 所有生产业务数据必须来自数据库/API，不得在前端、后台或后端生产路径硬编码店铺、商品、价格、图片、库存、卡密、客服二维码、收款码、渠道关系或 mock 支付结果。
3. 商户有自己的独立店铺分享地址，买家只能看到当前店铺，不展示其他商户入口。
4. 平台、一级、二级、三级是供货价差关系，不是返佣、返利、团队奖或邀请奖励。
5. 平台供货价只能给一级看到；二级只能看到一级给自己的转供价；三级只能看到二级给自己的转供价。
6. 支付口径已更新：后台收款绑定以 `docs/18-payment-binding-and-voucher-removal-plan.md` 为准。支付宝商户、腾讯/微信商户、e支付走接口回调/查单自动确认；个人支付宝只走人工确认；上传付款截图不再是主支付流程。

## 2. 卡密购买和提取的最终口径

这是当前最容易跑偏的地方，必须按以下规则实现。

1. 买家下单时邮箱是选填项，不是必填项，也不是唯一取码方式。
2. 商品后台配置是否需要购买密码；不是所有自动发码商品都必须有购买密码。
3. 启用购买密码的商品，下单时买家必须自己设置购买密码，可以是纯数字。
4. 服务端只能保存购买密码哈希，不能保存明文。
5. 后台确认收款或后续真实支付回调确认成功后，系统自动发码并生成可提取权益。
6. 买家必须能进入独立卡密提取页；该页只要求输入购买密码，不要求再次输入订单号。
7. 订单上下文必须来自安全跳转 token、订单查询凭证或登录态订单详情，不得让买家手工输入订单号来拼接口。
8. 买家填写邮箱时，系统在确认收款并发码后额外把激活码/卡密发送到邮箱；邮箱发码是补充通道，不替代独立提取页。
9. 买家未填写邮箱时，订单仍然可以创建、支付、确认收款和提取卡密。
10. 购买密码错误 3 次锁定 30 分钟。
11. 已支付且未退款的订单允许重复查看已发放卡密。
12. 退款完成后禁止继续查看卡密，也禁止邮件补发。

## 3. 当前实现偏差

以下不是新需求，是现有实现和既定口径之间的偏差。

1. H5 当前主要在订单详情内弹出提取卡密，不是独立提取页。
2. 当前用户提取接口是 `POST /api/user/orders/:orderNo/extract`，URL 仍依赖 `orderNo`；这不满足“提取页只输入购买密码”的最终口径。正确方向是使用安全提取 token 或订单查询凭证派生订单上下文。
3. 代码目前有 `buyerEmail` 保存和展示逻辑，但没有真实邮件服务、邮件发送日志、失败重试或人工补发闭环。
4. 商户自有商品可以提交，但商户 scoped 卡密池导入、查看、作废、补发、审计的闭环仍不完整；当前卡密导入主要是平台 admin 入口。
5. 商户如果要销售自有卡密商品，必须具备商户自己的库存/卡密管理权限，否则“商户自有虚拟卡产品”无法真正生产使用。
6. H5 首页和商品详情的移动端样式仍需按真实手机视口持续回归，避免样式修改破坏购买、详情、下单、提取等业务入口。

## 4. P0 必补任务

### 4.1 独立卡密提取页

负责人：frontend_dev + backend_worker，integration_tester 验收。

1. 后端新增安全提取凭证模型或复用订单查询凭证，凭证必须绑定订单、买家、店铺、过期时间和使用范围。
2. 后端提供不暴露订单号的提取接口，例如 `POST /api/user/extractions/:token` 或等价设计；请求体只包含 `extractionCode`。
3. H5 新增独立提取页；从确认成功页、订单详情、订单查询结果进入时携带安全 token。
4. 启用购买密码的商品，下单页必须展示购买密码输入；未启用购买密码的商品不能强制输入。
5. 提取页必须覆盖成功、错误次数、锁定、退款后禁看、重复查看、过期凭证和无权限状态。
6. 后台订单详情必须能看到提取日志、锁定状态和退款后禁看状态。

### 4.2 邮箱发码闭环

负责人：backend_worker + database_expert，integration_tester 验收。

1. 增加邮件发送配置，生产环境未配置时必须明确禁用或报错，不能静默假成功。
2. 增加邮件发送记录：订单、买家邮箱、模板、发送状态、错误原因、重试次数、发送时间。
3. 确认收款并发码后，如果买家填写邮箱，则发送激活码/卡密。
4. 未填写邮箱不得阻塞下单、确认收款或卡密提取。
5. 退款后不得再次发送或补发卡密邮件。
6. 后台提供失败重试和人工补发入口，并写审计日志。

### 4.3 商户自有卡密库存

负责人：database_expert + backend_worker + frontend_dev，integration_tester 验收。

1. 商户自己的自动发码商品必须支持商户 scoped 卡密导入。
2. 商户只能管理自己商品的卡密，不能查看平台商品、上级商品或其他商户商品的卡密明文。
3. 卡密明文查看、导出、作废、补发必须有单独权限和审计。
4. 平台可以审核、风控和仲裁商户自有卡密商品，但默认不替商户导入自有库存，除非平台后台以平台权限代操作并留痕。
5. 商户自有商品既可以是自动卡密，也可以是无卡密/人工交付商品；商品级履约方式必须由数据库配置。

### 4.4 文档和实现同步门禁

负责人：pm_architect + reviewer。

1. 每次样式修改后，必须回归业务入口：商品列表、商品详情、下单、收款方式、订单详情、提取页、售后。
2. 每次业务逻辑修改后，必须更新相关文档；旧文档无法更新时要标注历史参考或删除。
3. reviewer 最终门禁必须检查本文列出的偏差是否已关闭。
4. 硬编码扫描必须覆盖 `apps/h5`、`apps/admin`、`apps/api` 和生产会加载的共享包；测试 fixture、Prisma seed 和文档示例除外。

## 5. 不允许再误解的点

1. “邮箱接收卡密”不是替代购买密码提取。
2. “订单详情弹窗提取”不是最终独立提取页。
3. “接口 URL 带订单号”不等于“提取页只输入购买密码”。
4. “平台 admin 能导入卡密”不等于“商户能管理自己的卡密库存”。
5. “隐藏按钮”不等于后端权限完成；API 响应和导出也必须隔离。
6. “种子商品存在”不等于生产硬编码；生产运行必须从数据库读取，种子只用于初始化。
