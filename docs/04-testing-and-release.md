# 测试验收与上线清单

本文是当前开发依据之一，覆盖平台自营、单层渠道、受控两级渠道、H5、小程序和支付占位的测试上线口径。

## 1. 测试原则

1. 测试重点不是页面是否能打开，而是交易、履约、退款、结算、对账和数据隔离是否正确。
2. 所有金额以后端订单快照、退款快照、结算明细和 ledger 为准。
3. 当前支付账号未开通阶段，使用 mock 支付验证完整状态机；真实支付账号开通后再做小额真实支付和退款。
4. 必须覆盖三类销售路径：`platform_self_operated`、`single_agent`、`two_tier`。
5. 禁止出现三级渠道、返佣、团队奖、邀请奖励、代理等级和按招募人数或团队订单计算收益。

## 2. 核心链路测试

### 2.1 平台自营购买链路

测试步骤：

1. 平台配置平台自营店。
2. 平台上架自营商品并设置自营售价。
3. 用户从小程序或 H5 进入平台自营店。
4. 用户下单并完成 mock 支付。
5. 平台履约。
6. 用户查看权益。
7. 平台查看自营订单、收入、履约成本和对账。

验收标准：

1. 订单归属为 `platform_self_operated`。
2. 平台自营订单不生成渠道结算。
3. 订单快照记录最终售价、履约成本、支付通道费和平台自营毛收益。
4. 渠道商不能查看平台自营经营明细。

### 2.2 单层渠道购买链路

验收标准：

1. 订单归属到正确渠道商和店铺。
2. 用户支付金额等于渠道售价。
3. 平台供货价、服务费、渠道差价收益计算正确。
4. 渠道商只能看到自己的订单、收益和结算。

### 2.3 受控两级渠道购买链路

验收标准：

1. 未授权一级不能创建二级渠道商。
2. 二级必须审核通过并缴纳保证金后才能销售。
3. 二级不能继续创建三级渠道。
4. 订单快照固化平台供货价、一级转供价、最终售价、一级差价、二级差价。
5. 两级渠道结算使用 `order_id + settlement_role` 唯一，一级和二级可分别结算但同一角色不得重复结算。

## 3. H5 与小程序测试

1. 小程序店铺可访问、下单、查看订单和权益。
2. H5 店铺可在微信外浏览器打开。
3. H5 与小程序共用同一后端商品、订单、履约、售后数据。
4. H5 用户身份与小程序用户身份隔离，未合并身份前不得串单。
5. 用户端不得展示供货价、转供价、服务费、渠道差价、结算金额和风控细节。

## 4. 支付测试

当前阶段：

1. mock 支付成功。
2. mock 支付取消或失败。
3. mock 支付回调重复、延迟、金额不一致。
4. mock 支付不得绕过订单、履约、退款、结算和 ledger。

真实账号开通后：

1. 微信小程序内：微信小程序支付，前端调用 `wx.requestPayment`。
2. 微信内 H5：微信 JSAPI 支付。
3. 微信外手机浏览器 H5：微信 H5 支付。
4. 普通手机浏览器：支付宝手机网站支付。
5. 小程序内不设计支付宝支付入口。

验收标准：

1. 支付金额必须等于后端订单金额。
2. 支付回调和退款回调必须验签和幂等。
3. 支付结果不得只依赖前端返回或 H5 跳转。
4. 支付渠道字段能区分 `wechat_miniprogram`、`wechat_h5_jsapi`、`wechat_h5`、`alipay_wap`、`mock`。

## 5. 退款与售后测试

1. 平台自营退款不生成一级/二级追扣，进入平台退款、成本和 ledger 对账。
2. 单层渠道退款按平台、代理、用户或混合责任拆分。
3. 两级渠道退款按平台、一级、二级、用户或混合责任拆分。
4. 退款金额不得超过用户实付金额。
5. 退款中订单不能进入结算。
6. 已结算渠道订单退款必须生成追扣。

## 6. 结算与对账测试

1. 平台自营订单不进入代理/渠道打款结算，进入平台收入、履约成本、支付通道费、退款和 ledger 对账。
2. 单层渠道订单保持 `settlement_items.order_id` 唯一。
3. 两级渠道订单保持 `settlement_items(order_id, settlement_role)` 唯一。
4. 履约成功后默认 T+1 进入可结算。
5. 退款中、投诉中、风控冻结、履约失败订单不得结算。
6. 结算单、人工打款文件、追扣记录必须可追溯。

## 7. 上线前清单

1. 平台自营店下单、履约、退款和对账跑通。
2. 单层渠道下单、履约、退款和结算跑通。
3. 两级渠道授权、二级入驻、转供价、下单、退款、一级/二级结算跑通。
4. H5 店铺可公网访问，HTTPS 和备案准备完成。
5. 小程序合法服务器域名配置完成。
6. PostgreSQL、Redis、对象存储、日志、备份和监控方案确认。
7. mock 支付在生产环境强制关闭。
8. 真实支付账号、证书、密钥和回调域名准备后，补真实支付和真实退款灰度。

## 7.1 当前本地 P0 验收状态

当前本地 P0 以 mock 支付和 demo 身份完成业务闭环验证，覆盖：

1. 平台自营店：商品展示、下单、mock 支付、履约、自营收入/成本对账，不生成渠道结算。
2. 单层渠道店：独立店铺、客服二维码、选品定价、最低限价、订单、售后、退款、T+1 结算。
3. 受控两级渠道：一级授权、二级关系、转供价、二级下单、供货链快照、一级/二级分别结算。
4. H5 店铺：微信外浏览器店铺、下单、mock 支付、订单列表、售后申请、客服二维码。
5. 后台管理：代理审核、保证金、商品、自有商品审核、平台自营商品、渠道关系/转供价、订单、履约、售后退款、结算、风控、审计、账务流水、支付配置占位。

生产上线前仍必须替换 demo 认证、接入 PostgreSQL/Prisma 事务服务、配置 Redis/对象存储/HTTPS/备案/监控，并在支付账号开通后补真实支付和真实退款验签链路。

## 8. 回滚或暂停标准

1. 出现金额计算错误、重复入账、重复履约、结算重复或退款超额。
2. 出现用户、渠道商、平台自营数据越权。
3. 二级渠道能继续创建三级渠道。
4. 平台自营订单错误生成渠道结算。
5. mock 支付在生产环境可用。
6. 支付或退款渠道流水无法对账。
