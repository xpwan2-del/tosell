# 系统整体架构

本文是 P0 架构依据。当前项目是 H5 独立虚拟商品小店系统，不再以微信小程序作为交付端。系统由平台后台、商户后台、H5 买家店铺前台、API 服务、PostgreSQL 数据库、对象存储、任务/缓存、审计和监控组成。

## 1. 系统定位

本项目不是纯前端页面，而是交易、库存、卡密、收款、履约、售后、供货应付/服务费对账、风控、审计一体的财务型业务系统。

P0 真实闭环：

后台手工创建一级商户或商户入驻申请页 -> 输入邀请码注册商户 -> 平台审核 -> 后台确认保证金 -> 开通独立店铺 -> 绑定客服和自己的收款通道 -> 选择平台商品或提交自有商品审核 -> 设置售价/转供价 -> 买家从 H5 店铺下单 -> 商户自收款/系统确认收款 -> 自动发码或手工交付 -> 售后退款 -> 供货应付/服务费/追扣对账 -> 审计。

系统必须坚持：

1. 商户审核通过并由后台确认保证金后才能正式销售、选品上架、代理平台商品或代理上级商品。
2. 每个商户拥有独立店铺、店铺链接、商品列表、客服二维码、收款通道、订单、收益和供货应付/服务费对账记录。
3. 平台商品有供货价、最低销售价、建议销售价、履约规则和售后规则。
4. 商户可自由定价，但后端必须校验售价不得低于最低销售价和自身上游供货价。
5. 商户收益来自售价与上游供货价之间的差价，扣除平台 0.5% 服务费后形成预计收益，不是返佣。
6. 商户自有商品必须平台审核通过后才能售卖。
7. 订单、退款、供货应付、服务费、保证金、追扣和收款确认必须可追溯。
8. P0 商户使用自己的收款码/链接收款，不做平台代收后 T+1 给商户打款，也不做商户自助提现。
9. 平台可以开设平台自营店直接销售，平台自营订单不参与渠道供货链。
10. 受控代理价差供货最多允许到三级：平台供货 -> 一级商户 -> 二级商户 -> 三级商户 -> 消费者。

## 2. 禁止边界

系统不得设计、实现或预留以下能力：

1. 四级渠道或更多层级。
2. 下级返佣、推广返佣、按比例返利。
3. 团队业绩奖、团队排名奖、平级奖、领导奖。
4. 邀请奖励、拉人奖励、招募人数奖励。
5. 代理等级、团队等级、按等级差异返利。
6. 以招募人数、团队 GMV、团队订单计算收益。

数据库、API、页面、导出报表和运营后台中不得出现 `commission_rate`、`downline_commission`、`team_performance`、`invite_reward`、`agent_level` 等分销语义字段。邀请码只能用于商户归属和渠道关系绑定，不能用于奖励计算。

## 3. 端与角色

### 3.1 H5 买家店铺前台

包含店铺首页、商品列表、搜索/分类、商品详情、下单页、收款码/跳转支付页、订单查询、订单详情、卡密提取页、售后申请和客服入口。

买家前台只能展示最终售价、实付金额、订单状态、履约状态、权益信息和售后信息。不得展示平台供货价、一级转供价、二级转供价、服务费、商户收益、供货应付、冻结金额和风控细节。

### 3.2 商户后台

包含入驻申请、邀请码注册、审核状态、保证金状态、店铺配置、客服二维码、收款通道、平台商品选品、自有商品提交、定价、商户自有卡密库存、订单、售后、收益、供货应付/服务费账单、追扣。商户卡密库存必须通过商户 scoped 权限管理，不得复用平台 admin 卡密接口越权操作。

一级商户可以管理自己开放给二级的商品和一级转供价。二级商户可以管理自己开放给三级的商品和二级转供价。三级商户只能管理自己的店铺和销售，不能创建下级。

### 3.3 平台后台

包含商户审核、后台手工创建一级商户/店铺、初始账号密码生成、保证金、平台商品库、自有商品审核、收款通道审核、订单、履约、售后仲裁、供货应付、服务费账单、追扣、风控、审计、平台自营、渠道关系、邀请码、转供价和基础看板。

平台后台必须区分运营、财务和管理员权限。运营不能执行清算确认、保证金扣减和账务冲正；财务不能修改商品价格和商品审核；管理员负责 RBAC、高风险配置和最终权限管理。

### 3.4 后端服务

后端是金额、状态、权限、库存、卡密、收款、供货应付/服务费、账务和审计的唯一权威。前端提交的价格、服务费、收益、退款金额、清算金额只能作为请求参数或展示辅助，不能作为最终依据。

### 3.5 数据库与账务

数据库是财务追溯核心。订单必须固化快照；退款、供货应付、服务费、追扣、保证金扣减必须写流水；账务流水只能追加，不能原地覆盖金额。

## 4. 核心领域模块

1. Auth/RBAC：H5 买家会话、商户账号、平台后台账号、角色权限、数据范围、敏感操作授权。
2. Merchant/Shop：商户入驻、审核、保证金状态联动、店铺资料、客服二维码、冻结、禁用。
3. Product/MerchantProduct：平台商品库、商户选品、自有商品审核、上下架、库存、图片和详情。
4. VirtualInventory：平台和商户 scoped 卡密码池、库存扣减、自动发码、手工交付、商品级提取码配置、独立提取页 token/订单查询凭证、提取锁定、邮箱发码。
5. Pricing：最低限价校验、上游供货价校验、服务费计算、收益计算、金额取整、订单金额快照。
6. ChannelSupply：邀请码层级、渠道关系、商品开放、转供价、逐级价格隔离、供货链快照。
7. Order：订单创建、店铺归属固化、订单状态机、超时关闭、订单查询与权限隔离。
8. CollectionChannel：支付宝个人码、支付宝商户码/链接、微信个人码、微信商户码/链接、审核、启用、支付记录快照。
9. PaymentConfirmation：后台/商户确认收款、后续真实支付回调、金额校验、幂等入账、支付成功触发履约。
10. Fulfillment：自动/人工履约、履约尝试、权益发放、失败重试、补发、撤销、重复发放拦截。
11. Coupon：注册赠券、商品适用范围、有效期、作废、不可叠加。
12. AfterSale/Refund：售后申请、冻结供货应付/服务费账单、责任归属、退款审批、退款拆账。
13. Clearing/Reconciliation：供货应付、服务费账单、清算明细唯一约束、清算凭证、追扣对账；不做平台代收 T+1 打款。
14. Deposit：保证金账户、缴纳确认、扣减、冻结、退还、余额不足限制。
15. RiskFreeze：订单冻结、店铺冻结、限制供货/清算、禁用、解冻后重新判断清算资格。
16. Ledger/Audit：不可变账务流水、审计日志、冲正流水、敏感操作留痕。
17. Reconciliation/Export：订单、收款、退款、供货应付、服务费、追扣、保证金、ledger 对账导出。
18. PlatformSelfOperated：平台自营店、自营商品、自营订单、自营履约、自营售后、自营收入成本和对账。

## 5. 主数据流

### 5.1 商户入驻与店铺开通

1. 熟人或内部渠道由平台后台手工创建一级商户和店铺，生成初始账号密码并记录交付状态。
2. 陌生商户通过邀请码提交入驻申请。
3. 后端根据邀请码确定一级、二级或三级归属；后台手工创建只能直接创建一级商户。
4. 运营审核申请。
5. 审核通过或后台手工创建后进入待缴保证金状态。
6. 财务/管理员在后台人工确认保证金收款，写保证金交易和账务流水。
7. 保证金满足要求后开通正式销售、选品上架和代理商品能力。
8. 未确认保证金前，商户可登录后台查看资料和保证金状态，但不得销售、不得选品上架、不得代理平台或上级商品。

### 5.2 商品、选品与逐级价格

1. 平台创建商品，配置平台供货价、最低销售价、建议销售价、履约规则和售后规则。
2. 一级看到平台开放商品、商品资料和平台给自己的供货价；只有保证金已确认时才能选品上架。
3. 一级选择商品上架到自己的店铺，也可以开放给二级并设置一级转供价。
4. 二级看到商品资料和一级给自己的一级转供价，但看不到平台供货价；只有保证金已确认时才能选品上架或代理一级商品。
5. 二级选择商品上架到自己的店铺，也可以开放给三级并设置二级转供价。
6. 三级看到商品资料和二级给自己的二级转供价，但看不到一级转供价和平台供货价；只有保证金已确认时才能选品上架销售。
7. 买家只看到最终销售价。

### 5.3 下单、收款与订单快照

1. 买家通过 `/s/:shopSlug` 进入单个 H5 店铺。
2. 买家选择商品、可选填写邮箱；仅当商品启用提取码时填写购买密码/提取码；选择优惠券。
3. 后端校验店铺、商品、库存、卡密、保证金确认状态、售价、收款通道和风控状态。
4. 后端计算实付金额、上游供货价、服务费和销售方预计收益。
5. 后端写入订单、订单项、金额快照、商品快照、店铺快照和供货链快照。
6. 订单支付记录绑定收款通道快照。
7. 确认收款后订单进入已支付并触发自动发码或人工交付。

供货链快照需要保存完整清算依据，但对不同角色返回时必须脱敏：

1. 平台可读取完整快照。
2. 一级可读取平台给自己的供货价、一级转供价、相关差价。
3. 二级不得读取平台供货价。
4. 三级不得读取平台供货价和一级转供价。

### 5.4 卡密履约

1. 确认收款或后续真实支付成功后创建履约任务。
2. 自动卡密商品扣减库存并生成权益凭证。
3. 手工交付商品展示客服入口并等待商户处理。
4. 提取码是否启用由商品后台配置；启用时独立提取页只校验买家购买密码/提取码，订单上下文来自安全 token、订单查询凭证或登录态订单详情，错误 3 次锁定 30 分钟。
5. 买家邮箱为选填；填写邮箱时确认收款发码后同步发送激活码/卡密。邮箱发码是补充通道，不替代独立提取页。
6. 退款完成后权益不可继续查看。

### 5.5 售后退款

1. 买家提交售后申请。
2. 订单进入退款中或售后冻结状态，排除供货清算。
3. 商户先处理，必要时平台介入。
4. 后台录入责任归属、退款金额、平台承担、一级承担、二级承担、三级承担和服务费处理。
5. 未清算订单扣减供货应付/服务费账单；已清算订单生成追扣。

### 5.6 供货清算与对账

1. 商户使用自己的收款码/链接收款，P0 不做平台代收后 T+1 给商户打款。
2. 定时任务按商户和窗口筛选可清算订单。
3. 条件包括确认收款、履约成功、无退款中、无投诉中、无风控冻结。
4. 多级订单使用 `order_id + clearing_role` 唯一，防止同一订单同一清算角色重复清算。
5. 财务确认供货应付、服务费和追扣账单；如线下清算发生，只回填清算凭证。
6. 系统写清算记录、账务流水和审计日志。

## 6. 外部依赖

1. H5 域名、API 域名、后台域名、HTTPS 和备案。
2. PostgreSQL 数据库。
3. Redis 或任务锁服务。
4. 对象存储：商品图片、客服二维码、收款二维码、售后凭证、清算凭证。
5. 邮件或短信服务：注册登录、订单通知、找回订单。
6. 后续真实支付：支付宝、微信商户支付或其他渠道的账号、密钥、证书和回调域名。
7. 日志、备份、监控和告警。

## 7. P0 必做

1. H5 买家端交易闭环。
2. 商户端入驻、保证金、店铺、选品、转供价、自有商品库存卡密、收款通道、订单、售后、供货清算。
3. 平台后台审核、商品、订单、履约、售后、供货清算、风控、审计。
4. 后端订单快照、金额计算、收款确认幂等、履约、售后、供货清算、追扣、保证金、权限、审计。
5. 数据库核心实体、金额字段、唯一约束、索引、账务流水、审计日志。
6. 逐级价格隔离专项测试。
