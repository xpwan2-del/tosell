## 1. 验收结论规则

整体结论只允许三种：

1. `PASS`：页面、API、数据库、权限、审计全部通过，没有 P0 阻断。
2. `CONDITIONAL_PASS`：当前非真实支付阶段可用，但存在明确外部依赖未接入，例如真实支付回调、真实邮件服务。
3. `FAIL`：任一 P0 场景失败，不能上线。

任何下列情况直接判定 `FAIL`：

1. 生产路径依赖固定店铺、固定商品、固定价格、固定收款码、固定卡密或 mock 支付结果。
2. 未交保证金的商户可以选品、上架、代理商品、创建可支付订单或配置下游转供价。
3. 二级商户能看到平台给一级的供货价。
4. 三级商户能看到平台供货价或一级给二级的转供价。
5. 三级商户可以发展四级。
6. H5 买家能看到其他商户店铺入口。
7. 自动卡密商品确认收款后不发码，或退款后仍能查看卡密。
8. 商户 A 能操作商户 B 的商品、订单、卡密、收款通道或售后。
9. 后台页面展示成功，但数据库没有对应记录。
10. 生产环境可用 demo header、mock payment 或小程序登录。
11. 商户后台页面使用 `/api/admin/*` 完成商户经营动作，而不是使用 merchant bearer 的 `/api/agent/*`。
12. 商户看板显示全平台数据、上级数据或错误汇总，而不是当前商户 scoped GMV、订单、退款、待结算、追扣和保证金。
13. 客户误操作导致重复下单、重复扣库存、重复发码、错误订单归属、优惠券重复核销或售后重复创建。
14. 登录、token、账号冻结、角色权限等认证边界可绕过。
15. 优惠券可重复使用、并发重复核销、过期/未生效仍可用。
16. 保证金扣减、冻结、退款、余额不足等非 paid 状态下仍可继续销售或转供。
17. 风控冻结店铺、商品、订单或商户后，H5 和后台仍可继续交易、履约或结算。
18. 导出或敏感查看未脱敏、无权限控制或无审计。
19. 修改密码、禁用账号、角色降权后，旧密码、旧 token 或旧权限仍能继续操作。
20. 部分退款、多次退款、重复审批或重复确认导致累计退款超过实付，或账务重复入账/扣账。
21. 结算单、打款确认、追扣、服务费或价差计算可以被重复生成、并发重复确认，或金额公式与订单快照不一致。
22. 店铺、商品、公告、客服码、收款链接、售后备注等可输入内容触发 XSS、`javascript:` 链接或超长内容破坏页面。
23. 手机号、邮箱、收款账号、退款凭证、卡密明文、数据库密钥或 token 在页面、导出、日志中未按角色脱敏。
24. 商品下架、收款通道停用、商户关闭、审核状态回退后，新订单仍可继续购买，或历史订单快照被错误覆盖。
25. 登录、注册、邀请码校验、订单查询、提取码尝试等公开接口没有限流/锁定策略，导致撞库、刷券或暴力提取。
26. 验收前未准备完整测试数据，尤其缺少自动卡密商品的可用/空库存/重复/非法卡密池，却仍宣称完成页面级验收。

## 1.1 PM 复核标红项

以下是 PM 复核后要求特别标红的 P0 项：

1. 商户自有卡密导入和自有商品自动发码是 P0 验收项。一级、二级、三级都必须能创建自己的虚拟卡产品；自动卡密商品必须能导入自己商品的卡密并在确认收款后自动发码。
2. 商户看板必须按商户权限范围计算，不只验证接口返回，还要核对页面数字和数据库/API 汇总一致。
3. 商户后台所有经营操作必须使用 merchant bearer 调 `/api/agent/*`。平台后台才可以使用 `/api/admin/*`。商户页面混用 admin API 一律阻断。
4. Neon production sequence 必须进程 exit 0，不能只看 summary 字段。
5. 后台验收必须使用不同层级商户账号分别登录测试，不能只用平台账号代操作。买家也必须分不同用户分别在一级、二级、三级店铺购买，再回到对应商户后台核对订单、发货、售后、看板和结算。
6. 第二轮 PM 遗漏审计 `job_ae583a23e71e` 已完成，要求补齐认证/session 负向、优惠券强边界、保证金全状态、RBAC、收款凭证、库存并发、风控冻结、导出审计和部署运维回归。
7. 第三轮 PM 遗漏审计 `job_bd2359ada74b` 已完成，要求补齐账号生命周期、部分退款/多次退款、结算并发幂等、金额精度和分账公式、内容安全/XSS、PII 脱敏、审核状态回退和公开接口限流。

当前结论：

1. 允许进入完整页面级生产验收。
2. 不允许直接对外生产上线。
3. reviewer 已确认第一轮结果不是最终上线 `PASS`。

当前阶段边界：

1. 真实支付平台绑定、跳转支付、真实支付回调和退款回调最后接入；当前不以真实支付平台作为上线前 P0。

### 2.6 禁止进入 production seed 的数据

以下数据只能存在于 test fixture、staging/e2e setup 或临时验收脚本，禁止写进 production seed：

1. 固定商品、固定价格、固定卡密、固定收款码、固定客服二维码。
2. 可登录的测试 admin、merchant、buyer 默认账号密码。
3. 模拟支付成功结果、mock refund callback、demo header 白名单。模拟支付只能用于本地/预发/受控验收，不得作为 production seed 或正式对外支付入口。
4. 固定邀请码、固定层级关系、固定商户链路。
5. 真实卡密明文、第三方密钥、JWT secret、数据库连接串。
6. 会污染真实财务的订单、退款、结算、保证金流水、ledger。
7. “测试订单”“快速下单”“一键发码”等生产可写测试入口。

8. 生产部署不能使用 `.env.example` 的开发默认值。生产必须显式配置 `APP_ENV=production`、`NODE_ENV=production`、`ALLOW_DEMO_AUTH=false`、`MOCK_PAYMENT_ENABLED=false`、`DATABASE_URL`、`AUTH_TOKEN_SECRET`。
