# 状态机与权限隔离

本文是 P0 开发依据，定义 H5 虚拟商品小店的商户、店铺、商品、收款、订单、卡密、售后、供货清算、保证金、风控状态机，以及平台、商户、买家的数据隔离规则。

## 1. 总原则

1. 状态流转以后端为准，前端只能展示后端枚举和 `display_status`。
2. 每次状态迁移必须校验当前状态、操作者角色、目标对象归属、风控限制和幂等键。
3. 影响金额、卡密、收款、履约、退款、供货清算、保证金、风控、权限的状态迁移必须写审计日志。
4. 商户侧 API 必须从认证主体派生 `merchant_id/shop_id`，不得信任前端传入的商户或店铺标识。
5. 买家侧 API 必须从登录主体派生 `user_id`，不得访问他人订单、卡密和权益。
6. 平台后台必须区分运营、财务、管理员权限。
7. 禁止任何围绕团队业绩、邀请奖励、代理等级、返佣比例的状态、权限和报表。
8. 渠道供货最多允许平台供货 -> 一级商户 -> 二级商户 -> 三级商户 -> 消费者，三级不得继续创建四级。
9. 商品资料可以向下级开放，越级供货价必须服务端隔离。
10. 生产运行代码不得依赖硬编码店铺、商品、商户、收款码、价格、客服或测试按钮。

## 2. 商户状态机

状态：

1. `draft`：资料草稿。
2. `pending_review`：审核中。
3. `rejected`：审核拒绝。
4. `pending_deposit`：审核通过，待缴保证金。
5. `active`：已开通。
6. `frozen`：冻结，限制销售或供货清算。
7. `disabled`：禁用，不允许新增销售。
8. `exit_observation`：退出观察期。
9. `exited`：已退出。

规则：

1. 商户提交资料：`draft -> pending_review`。
2. 平台后台手工创建熟人一级商户：直接创建一级商户、店铺和初始账号，状态进入 `pending_deposit`，并记录创建人和审计日志。
3. 平台审核拒绝：`pending_review -> rejected`，必须填拒绝原因。
4. 平台审核通过：`pending_review -> pending_deposit`。
5. 财务/管理员确认保证金：`pending_deposit -> active`。
6. 风控冻结：`active -> frozen`，必须写原因。
7. 禁用商户：`active/frozen -> disabled`。
8. 退出退还保证金前必须检查未完成订单、售后、投诉、冻结和追扣。
9. `pending_deposit`、`rejected`、`frozen`、`disabled` 状态不得销售、不得选品上架、不得代理平台或上级商品、不得配置转供价。

## 3. 邀请码与渠道关系状态

邀请码状态：

1. `active`
2. `used_up`
3. `expired`
4. `disabled`

规则：

1. 平台邀请码只能创建一级商户。
2. 一级邀请码只能创建该一级名下二级商户。
3. 二级邀请码只能创建该二级名下三级商户。
4. 三级商户不得创建邀请码。
5. 邀请码只用于归属和供货关系，不产生邀请奖励、拉人奖励或返佣。

渠道关系状态：

1. `pending_review`
2. `pending_deposit`
3. `active`
4. `frozen`
5. `closed`

规则：

1. 一个商户同一时间只能有一个有效上级。
2. 二级只能绑定一个有效一级，三级只能绑定一个有效二级。
3. 四级关系创建必须被后端拒绝并写安全事件。

## 4. 店铺状态机

状态：

1. `not_opened`
2. `configuring`
3. `open`
4. `frozen`
5. `disabled`

规则：

1. 商户未审核通过或未缴保证金时，店铺不得营业。
2. `open` 状态可展示商品并允许下单。
3. `frozen` 状态不得新增下单，可保留订单、售后和供货清算查询。
4. `disabled` 状态不得新增销售。
5. 店铺名称、公告、客服二维码、收款配置必须来自数据库/API，不得写死在前端。
6. 未确认保证金的店铺可以配置基础资料，但不得展示为可购买店铺。

## 5. 商品与转供价状态

平台商品状态：

1. `draft`
2. `active`
3. `inactive`
4. `risk_removed`

商户商品状态：

1. `draft`
2. `pending_review`
3. `rejected`
4. `approved`
5. `listed`
6. `delisted`
7. `risk_removed`

转供商品状态：

1. `active`
2. `suspended`
3. `closed`

规则：

1. 平台商品由后台运营维护，下架后不得新增选品或购买。
2. 商户自有商品必须先审核，审核通过后才能售卖。
3. 一级可向二级开放商品并设置一级转供价。
4. 二级可向三级开放商品并设置二级转供价。
5. 二级能看到商品资料和一级给自己的转供价，但不能看到平台给一级的供货价。
6. 三级能看到商品资料和二级给自己的转供价，但不能看到一级转供价和平台供货价。
7. 商品图片、详情、规格、售后规则、库存和卡密必须从数据库或对象存储读取，不得硬编码。
8. 商户保证金未确认时，不得选择平台商品、不得选择上级开放商品、不得新增商户商品上架、不得配置给下级的转供价。

## 6. 收款配置状态机

状态：

1. `draft`
2. `pending_review`
3. `active`
4. `rejected`
5. `disabled`

规则：

1. 商户可提交支付宝商户、腾讯/微信商户、e支付、个人支付宝。
2. 平台审核通过后才能在 H5 支付页展示。
3. 禁用、拒绝、过限额通道不得展示。
4. 订单支付记录必须保存收款配置快照。
5. 收款码、账号名称和跳转链接必须来自数据库/API，不得硬编码。

## 7. 订单与收款状态机

订单状态：

1. `pending_payment`
2. `pending_payment_confirmation`
3. `paid`
4. `fulfilling`
5. `fulfilled`
6. `fulfillment_failed`
7. `after_sale_pending`
8. `refunding`
9. `refunded`
10. `closed`

规则：

1. 下单时后端校验店铺、商品、库存、卡密、保证金、价格、优惠券、收款配置和风控。
2. 个人支付宝订单进入 `pending_payment_confirmation`；支付宝商户、腾讯/微信商户、e支付订单等待回调/查单确认。
3. 回调/查单或个人支付宝有权限人工确认后，订单变为 `paid` 并触发履约。
4. 确认收款必须幂等，重复确认不得重复履约、重复发码或重复写账。
5. 多级订单必须固化完整供货链快照，但对不同角色返回时必须脱敏。
6. 历史订单归属不得因商品改价、商户状态变化或买家再次进入其他店铺而改变。

## 8. 卡密与履约状态机

履约状态：

1. `not_started`
2. `processing`
3. `success`
4. `failed`
5. `resent`
6. `revoked`

卡密状态：

1. `available`
2. `reserved`
3. `issued`
4. `refunded_locked`
5. `voided`

规则：

1. 自动卡密商品支付成功后扣减库存并发放卡密。
2. 手工交付商品支付成功后显示客服入口，不显示卡密。
3. 购买密码必须哈希存储。
4. 购买密码错误 3 次锁定 30 分钟。
5. 已支付且未退款订单允许重复查看卡密。
6. 退款完成后禁止继续查看卡密。
7. 提取页必须是独立页面，只输入购买密码，订单上下文来自支付成功跳转 token、登录态订单详情或安全订单查询凭证，不要求买家再次输入订单号。
8. 每次成功、失败、锁定和退款后拒绝都必须写入 `order_extract_logs`。

## 9. 优惠券状态机

状态：

1. `active`
2. `used`
3. `expired`
4. `voided`
5. `voided_after_refund`

规则：

1. 平台可配置新用户注册赠券。
2. 优惠券可限制适用商品、店铺、有效期和状态。
3. 单笔订单只能使用一张优惠券。
4. 优惠金额以后端计算为准。
5. 平台券属于平台补贴，不影响平台供货价、一级转供价、二级转供价和各级价差结算基准。
6. 退款完成后，已使用优惠券进入 `voided_after_refund`，不退回买家券包。
7. 优惠券模板、发放、使用、作废和退款后作废都必须可审计。

## 10. 售后退款状态机

状态：

1. `pending`
2. `merchant_processing`
3. `platform_intervening`
4. `refund_approved`
5. `refunding`
6. `refunded`
7. `rejected`
8. `cancelled`

责任归属：

1. `platform`
2. `merchant`
3. `first_tier`
4. `second_tier`
5. `third_tier`
6. `user`
7. `mixed`

规则：

1. 买家提交售后后，订单供货清算必须冻结。
2. 平台介入后可以做最终裁决。
3. 退款金额累计不得超过用户实付金额。
4. 售后单必须记录责任归属、退款金额、平台承担、商户承担、一级承担、二级承担、三级承担、服务费退还金额和服务费承担方。
5. 退款完成后自动卡密权益进入不可查看状态。

## 11. 供货清算与保证金状态

清算状态：

1. `pending`
2. `frozen`
3. `clearable`
4. `clearing`
5. `cleared`
6. `clawback_pending`
7. `clawed_back`

进入可清算条件：

1. 已确认收款或真实支付成功。
2. 履约成功。
3. 未退款，未退款中。
4. 未被投诉或风控冻结。
5. 商户自收款场景只生成供货应付、服务费和追扣清算，不做平台代收 T+1 打款。

保证金状态：

1. `pending_payment`
2. `paid`
3. `partially_deducted`
4. `frozen`
5. `refund_reviewing`
6. `refunded`
7. `insufficient`

规则：

1. 多级订单使用 `clearing_items(order_id, clearing_role)` 唯一防重复。
2. 平台自营订单不进入渠道供货清算。
3. 保证金缴纳、扣减、冻结、退还必须写 `deposit_transactions` 和 `ledger_entries`。

## 12. 权限与价格可见矩阵

| 角色 | 可访问数据 | 可见价格 | 禁止内容 |
| --- | --- | --- | --- |
| 买家 | 当前店铺商品、自己的订单/卡密/售后 | 最终售价、优惠后实付 | 供货价、转供价、差价、服务费、供货清算、其他店铺切换 |
| 一级商户 | 自己店铺、订单、收款配置、下级授权范围 | 平台给自己的供货价、自己给二级的转供价、自己售价 | 无关商户数据、返佣、团队收益 |
| 二级商户 | 自己店铺、订单、收款配置、三级授权范围 | 一级给自己的转供价、自己给三级的转供价、自己售价 | 平台给一级的供货价、一级其他下级 |
| 三级商户 | 自己店铺、订单、收款配置 | 二级给自己的转供价、自己售价 | 平台供货价、一级转供价、新增下级 |
| 运营 | 商户、商品、订单、履约、售后、风控 | 按岗位授权 | 清算确认、改账 |
| 财务 | 保证金、供货清算、服务费、对账、追扣 | 财务授权范围 | 改商品、绕过售后裁决 |
| 管理员 | 全局管理数据 | 全局授权范围 | 绕过审计直接改账 |

## 13. 无硬编码红线

以下内容在生产运行代码中必须来自数据库、对象存储、环境配置或后端 API，禁止写死：

1. 店铺 ID、商户 ID、商品 ID、订单 ID、渠道关系 ID。
2. 商品标题、图片、详情、规格、售价、库存、卡密。
3. 平台供货价、一级转供价、二级转供价、最低限价、优惠券金额。
4. 客服微信、客服二维码、收款二维码、支付跳转链接。
5. 默认测试店铺切换、固定预览入口、固定后台快捷操作。
6. mock 支付按钮、mock 回调入口和固定确认收款结果。

允许保留：

1. 单元测试、集成测试和种子数据中的示例值。
2. `.env.example` 中的示例配置。
3. 文档里的业务说明示例。

## 14. 审计要求

必须写审计日志的动作：

1. 商户审核通过/拒绝。
2. 店铺冻结、解冻、禁用。
3. 商品上下架、风险下架、后台价格调整。
4. 收款配置审核、启用、禁用。
5. 邀请码创建、禁用和使用。
6. 一级/二级转供价调整。
7. 订单确认收款、冻结、解冻、备注。
8. 人工履约、补发、撤销。
9. 售后裁决、退款金额确认、责任归属变更。
10. 供货清算单生成、确认、清算凭证回填。
11. 保证金缴纳确认、扣减、冻结、退还。
12. 权限变更和角色授权。

审计日志必须包含：操作者、角色、动作、目标、操作前值、操作后值、原因、request_id、IP、时间。
