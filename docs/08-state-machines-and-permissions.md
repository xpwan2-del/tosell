# 状态机与权限隔离

本文是开发依据文档，定义代理、渠道、店铺、商品审核、订单、支付、履约、售后退款、结算、保证金、风控状态机，以及用户、代理/渠道、运营、财务、管理员的数据隔离和操作权限。

## 1. 总原则

1. 状态流转以后端为准，前端只能展示后端枚举和后端返回的 `display_status`。
2. 每次状态迁移必须校验当前状态、操作者角色、目标对象归属、风控限制和幂等键。
3. 影响金额、履约、退款、结算、保证金、风控、权限的状态迁移必须写审计日志。
4. 代理侧 API 必须从认证主体派生 `agent_id/shop_id`，不得信任前端传入的代理或店铺标识。
5. 用户侧 API 必须从登录主体派生 `user_id`，不得访问他人订单和权益。
6. 平台后台必须区分运营、财务、管理员权限。
7. 禁止任何围绕团队业绩、邀请奖励、代理等级和返佣比例的状态、权限和报表。
8. 受控两级渠道只允许平台 -> 一级渠道商 -> 二级渠道商 -> 消费者，二级不得继续创建三级渠道。

## 2. 代理状态机

状态：

1. `draft`：资料草稿。
2. `pending_review`：审核中。
3. `rejected`：审核拒绝。
4. `pending_deposit`：审核通过，待缴保证金。
5. `active`：已开通。
6. `frozen`：冻结，限制销售或结算。
7. `disabled`：禁用，不允许新增销售。
8. `exit_observation`：退出观察期。
9. `exited`：已退出。

核心迁移：

1. 代理提交资料：`draft -> pending_review`，代理可操作。
2. 运营审核拒绝：`pending_review -> rejected`，运营可操作，必须填拒绝原因。
3. 运营审核通过：`pending_review -> pending_deposit`，运营可操作。
4. 财务确认保证金：`pending_deposit -> active`，财务可操作。
5. 风控冻结：`active -> frozen`，运营或管理员可操作，必须写原因。
6. 禁用代理：`active/frozen -> disabled`，管理员或授权运营可操作。
7. 申请退出：`active/frozen -> exit_observation`，需检查未完成订单、退款、投诉、冻结和追扣。
8. 观察期完成：`exit_observation -> exited`，财务或管理员确认。

## 3. 店铺状态机

状态：

1. `not_opened`：未开通。
2. `configuring`：配置中。
3. `open`：营业中。
4. `frozen`：冻结。
5. `disabled`：禁用。

规则：

1. 代理未审核通过或未缴保证金时，店铺不得营业。
2. `open` 状态可展示商品并允许下单。
3. `frozen` 状态不得新增下单，可保留订单、售后和结算查询。
4. `disabled` 状态不得新增销售，用户入口给出不可售提示。
5. 冻结、解冻、禁用必须写审计。

## 3.1 渠道授权状态机

一级渠道能力状态：

1. `not_requested`：未申请。
2. `pending_review`：申请审核中。
3. `authorized`：已授权，可创建二级渠道关系。
4. `rejected`：审核拒绝。
5. `suspended`：暂停授权。
6. `closed`：关闭授权。

核心规则：

1. 未授权一级不得创建二级渠道商。
2. 授权、拒绝、暂停、关闭都必须写审计。
3. 关闭授权后不得新增二级关系，历史订单和结算仍按订单快照处理。

二级渠道关系状态：

1. `pending_review`：二级申请审核中。
2. `pending_deposit`：审核通过待缴保证金。
3. `active`：已开通。
4. `frozen`：冻结。
5. `closed`：关闭。

核心规则：

1. 二级必须审核通过并缴纳保证金后才能销售。
2. 二级只能绑定一个有效一级渠道商。
3. 二级不得继续创建三级渠道关系。

## 4. 商品状态机

### 4.1 平台商品

状态：

1. `draft`
2. `active`
3. `inactive`
4. `risk_removed`

规则：

1. 平台商品由后台运营维护。
2. 下架后代理不得新增上架或新增购买。
3. 商品改价不得影响历史订单快照。
4. 风险下架必须记录原因。

### 4.2 代理商品与自有商品审核

状态：

1. `draft`
2. `pending_review`
3. `rejected`
4. `approved`
5. `listed`
6. `delisted`
7. `risk_removed`

规则：

1. 平台商品选品可直接进入代理商品上架流程，但售价必须通过最低限价校验。
2. 代理自有商品必须先审核，`approved/listed` 后才能售卖。
3. 审核拒绝必须记录原因。
4. 风险下架后不得购买。
5. 代理不能审批自己的商品。

## 5. 订单状态机

状态：

1. `pending_payment`：待支付。
2. `paid`：已支付。
3. `fulfilling`：履约中。
4. `fulfilled`：履约成功。
5. `fulfillment_failed`：履约失败。
6. `after_sale_pending`：售后中。
7. `refunding`：退款中。
8. `refunded`：已退款。
9. `closed`：已关闭。

规则：

1. `pending_payment` 超时可关闭。
2. 支付成功后进入 `paid/fulfilling`。
3. 履约失败不得进入可结算。
4. 售后和退款中必须冻结结算。
5. 已退款订单不得重复申请同一笔全额退款。
6. 历史订单归属不得因用户再次进入其他店铺、商品改价、代理状态变化而改变。
7. 两级渠道订单必须固化一级渠道、二级渠道、平台供货价、一级转供价、最终售价、一级差价、二级差价和渠道授权快照。

## 6. 支付状态机

状态：

1. `unpaid`
2. `paying`
3. `paid`
4. `failed`
5. `cancelled`
6. `expired`

规则：

1. 支付成功以后端收到并验签通过的回调为准。
2. 回调金额必须等于订单应付金额。
3. 支付回调必须幂等，重复回调不得重复履约、重复入账或重复写收益。

## 7. 履约状态机

状态：

1. `not_started`
2. `processing`
3. `success`
4. `failed`
5. `resent`
6. `revoked`

规则：

1. 每次履约尝试必须有 `fulfillment_attempts`。
2. 成功履约生成 `entitlements`。
3. 重试和补发必须使用幂等键，避免重复发放同一权益。
4. 权益撤销必须记录撤销原因和操作人。
5. 履约失败订单不得进入可结算。

## 8. 售后退款状态机

状态：

1. `pending`
2. `agent_processing`
3. `platform_intervening`
4. `refund_approved`
5. `refunding`
6. `refunded`
7. `rejected`
8. `cancelled`

责任归属：

1. `platform`：平台责任。
2. `agent`：单层代理责任。
3. `first_tier`：两级渠道中的一级渠道商责任。
4. `second_tier`：两级渠道中的二级渠道商责任。
5. `user`：用户原因。
6. `mixed`：混合责任。

规则：

1. 用户提交售后后，订单结算必须冻结。
2. 平台介入后可以做最终裁决。
3. 退款金额累计不得超过用户实付金额。
4. 售后单必须记录责任归属、退款金额、平台承担、代理承担、一级承担、二级承担、服务费退还金额和服务费承担方。
5. 单层订单可使用 `agent`；两级渠道订单必须明确为 `first_tier`、`second_tier` 或 `mixed`，不得只写模糊的 `agent`。
5. 未结算订单退款扣减待结算收益；已结算订单退款生成追扣。
6. 退款回调必须幂等。

## 9. 结算状态机

状态：

1. `pending`：待结算。
2. `frozen`：冻结中。
3. `settleable`：可结算。
4. `settling`：结算中。
5. `settled`：已结算。
6. `clawback_pending`：待追扣。
7. `clawed_back`：已追扣。

进入可结算条件：

1. 支付成功。
2. 履约成功。
3. 默认按履约成功时间 T+1。
4. 未退款。
5. 未处于退款中。
6. 未被投诉冻结。
7. 未被风控冻结。

规则：

1. 单层订单 `settlement_items.order_id` 必须唯一；两级渠道订单 `settlement_items(order_id, settlement_role)` 必须唯一。
2. 冻结订单不得进入结算单。
3. 已结算后发生退款、投诉赔付或违规扣罚，进入追扣流程。
4. V1 只做结算单和人工打款，不做代理自助提现。

## 10. 保证金状态机

状态：

1. `pending_payment`
2. `paid`
3. `partially_deducted`
4. `frozen`
5. `refund_reviewing`
6. `refunded`
7. `insufficient`

规则：

1. 未缴纳保证金不得正式销售。
2. 保证金缴纳、扣减、冻结、退还必须写 `deposit_transactions` 和 `ledger_entries`。
3. 扣减必须记录原因、金额、关联订单或事件、操作人和凭证。
4. 保证金不足时可限制结算、限制上架、暂停销售或暂停人工打款。
5. 退出退还前必须检查未完成订单、售后、投诉、冻结和追扣。

## 11. 风控状态机

状态：

1. `normal`
2. `order_frozen`
3. `shop_frozen`
4. `settlement_restricted`
5. `product_removed`
6. `disabled`

规则：

1. 风控目标可以是订单、店铺、商品、代理或结算。
2. 同一目标同类 active 冻结不得重复创建。
3. 冻结期间不可结算或不可下单，视冻结类型决定。
4. 解冻后重新按订单、退款、投诉、履约时间和保证金状态判断。
5. 风控动作必须写审计。

## 12. 权限矩阵

| 角色 | 可访问数据 | 可执行操作 | 禁止操作 |
| --- | --- | --- | --- |
| 普通用户 | 自己的订单、权益、售后 | 浏览代理店铺、下单、支付、查看权益、提交售后 | 查看他人订单、查看供货价、查看代理收益 |
| 代理 | 自己的店铺、商品、订单、售后、收益、结算、保证金记录 | 入驻、配置店铺、选品、定价、提交自有商品、查看收益和结算 | 查看其他代理数据、退款裁决、结算确认、人工打款、扣保证金、解除风控、自助提现 |
| 一级渠道商 | 自己的店铺、授权状态、二级渠道关系、已授权二级的转供商品、自己的订单和结算 | 申请二级渠道能力、配置可转供商品和转供价、查看一级渠道差价和结算 | 创建三级渠道、查看非绑定二级数据、按比例返佣、团队收益统计 |
| 二级渠道商 | 自己的店铺、商品、订单、售后、收益、结算、保证金记录 | 入驻、配置店铺、设置最终售价、销售商品、处理售后 | 创建三级渠道、修改一级转供价、查看一级其他数据、查看平台供货敏感配置 |
| 运营 | 代理、商品、订单、履约、售后、风控相关数据 | 代理审核、商品审核、订单备注、履约处理、售后处理、风控建议或冻结 | 人工打款、财务结算确认、修改财务流水 |
| 财务 | 保证金、结算、打款、对账、追扣 | 保证金确认、结算确认、人工打款回填、对账导出、追扣处理 | 修改商品价格、审核商品、绕过售后裁决退款 |
| 管理员 | 全局管理数据 | RBAC、系统配置、高风险操作审批、审计追踪 | 绕过审计直接改账 |

## 13. 状态迁移审计要求

必须写审计日志的动作：

1. 代理审核通过/拒绝。
2. 店铺冻结、解冻、禁用。
3. 商品上下架、风险下架、后台价格调整。
4. 自有商品审核通过/拒绝。
5. 订单冻结、解冻、备注。
6. 人工履约、补发、撤销。
7. 售后裁决、退款金额确认、责任归属变更。
8. 结算单生成、确认、人工打款回填。
9. 保证金缴纳确认、扣减、冻结、退还。
10. 追扣创建和扣减。
11. 权限变更和角色授权。
12. 一级渠道授权、暂停、关闭。
13. 二级渠道审核、冻结、关闭。
14. 一级转供价调整。
15. H5 域名、支付渠道配置变更。

审计日志必须包含：操作者、角色、动作、目标、操作前值、操作后值、原因、request_id、IP、时间。

## 14. 安全与数据隔离要求

1. 代理端查询不允许以前端传入的 `agent_id/shop_id` 作为可信条件。
2. 用户端查询不允许以前端传入的 `user_id` 作为可信条件。
3. 后台列表需要 RBAC 和数据范围控制。
4. 导出接口必须校验角色权限并写审计。
5. 篡改价格、篡改服务费、篡改代理收益、篡改退款金额必须被后端拒绝或重新计算。
6. 禁用或冻结店铺继续下单必须失败。
7. 绕过最低限价直接调用 API 必须失败并记录安全事件。
8. 所有敏感状态迁移必须有失败重试和幂等保护。
9. H5 用户端必须从登录态派生 `h5_user_id`，不得信任前端传入用户标识。
10. 二级渠道接口必须校验其所属一级渠道关系，不得通过前端参数访问其他一级或二级数据。

## 15. V1 必做与后置

V1 必做：

1. 本文列出的全部核心状态机。
2. 用户、代理、运营、财务、管理员权限隔离。
3. 代理数据强隔离。
4. 后台敏感操作审计。
5. 支付、退款、履约、结算、追扣、保证金交易幂等。

后置：

1. 自动风控评分。
2. 更细的审批流。
3. 自动提现权限。
4. 多供应商结算状态。
5. 高级 BI 权限模型。
