# 后台功能实现计划与目标

本文定义 ToSell 后台后续改造的目标、功能边界、模块结构、实施顺序和验收标准。它不是 UI 美化清单，而是把平台后台和商户后台从“数据库表格集合”改造成可日常运营的工作台。

依据文档：

1. `docs/01-product-requirements.md`
2. `docs/02-business-rules.md`
3. `docs/16-page-level-production-acceptance-test-plan.md`
4. `README.md`

## 1. 总目标

后台改造完成后，平台运营、平台财务、平台管理员、一级/二级/三级商户都能在后台完成自己的日常任务，并且页面、API、数据库、审计留痕一致。

目标不是“把所有接口都摆出来”，而是让使用者一眼知道：

1. 今天要处理什么。
2. 每条记录现在是什么状态。
3. 下一步能做什么。
4. 操作会影响谁、影响多少钱、是否会写入审计。
5. 出错后该怎么恢复或继续处理。

## 2. 核心成功标准

后台上线验收必须满足以下标准：

1. 首页只显示待办、异常、风险和核心经营数据，不堆全量表格。
2. 商品、订单、收款、退款、卡密、结算、商户等关键对象都能从列表进入详情页或详情抽屉。
3. 列表默认隐藏无意义 ID，优先展示中文名称、缩略图、状态、金额、归属、下一步动作。
4. 金额在后台统一按“元”展示和输入，后端仍以“分”存储，提交前由前端或 API 层转换并校验。
5. 表单字段必须使用运营可理解的中文名称，不直接暴露工程字段名。
6. 不让用户手填可选择的 ID；商品、商户、店铺、订单、优惠券、收款方式等必须优先使用搜索、选择器或从列表进入。
7. 高风险动作必须有确认弹窗、处理中状态、成功/失败反馈和审计说明。
8. 商户后台所有经营操作必须使用 merchant bearer 调 `/api/merchant/*`；平台后台才使用 `/api/admin/*`。
9. 二级、三级商户后台不能泄露上游不该看到的价格、利润、卡密、订单和结算信息。
10. 每个改造模块必须通过页面级验收：页面操作、API 响应、数据库记录、审计日志四层一致。

## 3. 非目标

本轮后台改造不做以下事项：

1. 不新增四级商户或更多层级。
2. 不做返佣、分销、团队业绩奖、拉人奖励。
3. 不做在线客服、工单系统或客服坐席后台，只维护微信/QQ 联系方式、二维码和客服说明。
4. 不做商户自助提现。
5. 不把真实支付宝/微信商户号联调作为阻塞项；但后台必须具备配置、验签、查单、回调准备能力。
6. 不允许为方便演示在生产路径硬编码店铺、商品、价格、库存、卡密、收款码或 mock 支付结果。
7. 不把后台继续拆成“每张数据库表一个页面”的形式。

## 4. 使用者与权限视角

### 4.1 平台运营

主要任务：

1. 审核商户和商户自有商品。
2. 创建和维护平台商品。
3. 管理平台自营店和自营商品。
4. 处理订单履约、售后、退款仲裁。
5. 冻结异常订单、商品、店铺或商户。

平台运营后台必须突出“待处理事项”和“下一步动作”。

### 4.2 平台财务

主要任务：

1. 确认保证金。
2. 查看服务费、供货应付、追扣和账务流水。
3. 处理退款责任拆分和人工退款确认。
4. 审核敏感收款配置和支付异常。

平台财务后台必须突出金额、状态、凭证、审计和幂等风险。

### 4.3 平台管理员

主要任务：

1. 管理后台账号、角色和权限。
2. 管理系统级配置。
3. 查看审计日志和敏感操作记录。
4. 处理紧急禁用、冻结和风险恢复。

平台管理员后台必须突出权限边界和敏感操作审计。

### 4.4 商户

主要任务：

1. 配置店铺资料和客服方式。
2. 配置自己的收款方式。
3. 选择平台或上级开放的商品。
4. 设置自己的销售价和展示内容。
5. 提交自有商品审核。
6. 管理自己的订单、发货、售后、卡密和收益。
7. 一级/二级商户管理下级供货价和邀请码。

商户后台必须按当前商户权限范围展示数据，不显示全平台信息，不泄露越级价格。

## 5. 后台信息架构

后台一级模块建议统一为以下结构。平台后台和商户后台可以共用壳，但菜单、权限、数据范围和可执行动作必须按身份裁剪。

### 5.1 工作台

目标：只回答“今天要处理什么”。

功能：

1. 待审核商户数。
2. 待确认保证金数。
3. 待审核自有商品数。
4. 待确认收款订单数。
5. 待发货订单数。
6. 支付异常数。
7. 售后/退款待处理数。
8. 库存告警商品数。
9. 风控冻结/异常数。
10. 待结算或待追扣事项。

验收：

1. 每张待办卡都能进入对应筛选后的列表。
2. 数据来自 API，不使用前端硬编码。
3. 平台账号看到全局待办；商户账号只看到自己权限内待办。

### 5.2 商品经营

目标：把“平台商品源、自营上架、商户代理、自有商品、卡密池”拆成清晰任务。

子模块：

1. 平台商品库。
2. 平台自营商品。
3. 商户代理商品。
4. 商户自有商品审核。
5. 商户自有商品管理。
6. 卡密池。

功能重点：

1. 平台商品创建、编辑、上下架、价格维护。
2. 平台自营店选择商品并设置自营售价。
3. 商户从可代理商品列表选择商品，不使用技术下拉框。
4. 商户可以修改本店展示内容，但不改变平台商品源。
5. 自有商品入口必须叫“新增自有商品 / 提交平台审核”。
6. 自动发码商品必须能导入、预检、查看库存、退款后禁看。

验收：

1. 商品列表显示缩略图、名称、类目、发货方式、库存、售价/成本、状态、下一步动作。
2. 金额输入使用元，提交后数据库仍保持分。
3. 商户未缴保证金时不能选品、上架、代理或配置转供价。
4. 商户只能管理自己的自有商品和自有卡密。
5. 代理上游商品不能查看上游卡密明文。

### 5.3 订单履约

目标：每个订单都能一眼看出支付状态、收款归属、发货状态、售后风险和下一步动作。

子模块：

1. 全部订单。
2. 待确认收款。
3. 待发货。
4. 已发货。
5. 异常订单。
6. 平台自营订单。
7. 商户订单。

功能重点：

1. 订单列表显示订单号、店铺、商户、商品、实付金额、支付方式、支付状态、发货状态、售后状态、下一步动作。
2. 个人支付宝/个人微信订单允许人工确认到账。
3. 商家支付/e支付订单必须依赖回调或查单确认，前端返回不能当成支付成功。
4. 自动卡密商品确认收款后自动发码。
5. 人工交付商品进入待人工交付。
6. 重复确认、并发确认、重复发码必须幂等。

验收：

1. 平台后台可查看全局订单。
2. M1/M2/M3 商户分别只能看到自己的订单和权限范围内供货相关记录。
3. 二级/三级订单不泄露越级价格。
4. 订单详情展示完整订单快照，不因商品后续改价而覆盖历史订单。

### 5.4 售后退款

目标：售后、退款、责任拆分和退款后禁看必须成为完整流程。

子模块：

1. 售后单。
2. 退款审批。
3. 责任拆分。
4. 人工退款确认。
5. 退款后卡密禁看。

功能重点：

1. 用户发起售后后，订单进入退款中或售后处理中。
2. 平台可仲裁责任归属：平台责任、商户责任、用户原因、混合责任。
3. 系统按用户实付金额退款，不允许累计退款超过实付。
4. 已发码订单退款后，买家不能再查看卡密。
5. 退款、追扣、保证金扣减、服务费冲正必须有 ledger 和审计。

验收：

1. 重复审批和重复退款不会重复入账或超额退款。
2. 商户只能处理自己订单的售后协助，不能裁决平台退款。
3. 平台后台可以看到责任拆分和审计记录。

### 5.5 商户平台

目标：商户入驻、保证金、邀请码、供货关系和下级授权成为可操作流程。

子模块：

1. 商户审核。
2. 手工创建一级商户。
3. 保证金管理。
4. 邀请码。
5. 下级供货授权。
6. 供货关系。
7. 转供价管理。

功能重点：

1. 熟人商户可由平台后台手工创建一级商户和店铺，生成初始账号密码。
2. 陌生商户通过邀请码注册并等待审核。
3. 审核通过后仍需确认保证金才能经营。
4. 一级可以授权二级，二级可以授权三级，三级不能继续发展四级。
5. 转供价不得低于上游成本和平台最低限价。

验收：

1. 未审核、审核拒绝、未缴保证金、已缴保证金、冻结商户的菜单和按钮状态正确。
2. 前端禁用只是提示，后端接口也必须拒绝越权操作。
3. 三级创建四级邀请码必须被拒绝。

### 5.6 财务收款

目标：收款配置、支付异常、余额、服务费、供货应付和追扣统一进入财务工作台。

子模块：

1. 收款配置。
2. 支付异常。
3. 余额充值。
4. 服务费配置。
5. 供货应付。
6. 追扣。
7. ledger 流水。
8. 对账。

功能重点：

1. 平台后台管理平台主店收款配置。
2. 商户自己的收款配置由商户后台管理，平台可审核和风控。
3. 支付密钥、私钥、证书只显示状态，不回显明文。
4. 收款配置卡片点击后必须回填表单。
5. 支付方式启用后，H5 只展示配置完整且启用的方式。
6. 默认支付手续费为 0；后台启用手续费后订单快照记录手续费比例和金额。

验收：

1. 收款配置写入 `collection_payment_configs`，不使用旧表。
2. 个人收款等待人工确认，商家支付依赖回调或查单。
3. 启用、停用、设为默认、测试签名、查看异常都写审计。
4. 金额、服务费、供货应付、追扣和 ledger 到分一致。

### 5.7 风控审计

目标：冻结、禁用、敏感查看、导出和审计成为统一安全后台。

子模块：

1. 风控冻结。
2. 店铺禁用。
3. 商品风险下架。
4. 订单冻结。
5. 卡密明文查看审计。
6. 导出审计。
7. 操作日志。

功能重点：

1. 冻结店铺、商品、订单、商户后，H5 和后台都不能继续对应高风险动作。
2. 卡密明文查看必须有权限控制和审计。
3. 导出必须脱敏，并记录导出人、时间、范围和原因。
4. 审计日志可按操作者、对象、动作、时间筛选。

验收：

1. 页面隐藏之外，直接调 API 也必须 401/403/404。
2. 敏感数据页面、导出、日志均按角色脱敏。
3. XSS、`javascript:` 链接和超长内容不能破坏后台页面。

### 5.8 店铺配置

目标：店铺资料、公告、客服二维码和售后说明可维护，并同步到 H5。

子模块：

1. 平台自营店。
2. 商户店铺资料。
3. 客服联系方式。
4. 公告。
5. 售后说明。

验收：

1. 后台修改店铺名、公告、客服二维码、微信号、QQ 号后，H5 刷新可同步展示。
2. 不出现在线客服、工单系统或坐席后台入口。
3. 图片和二维码链接必须校验格式，不能注入脚本。

## 6. 实施阶段

### 阶段 0：目标和门禁

交付：

1. 本文档确认。
2. 后台改造验收 checklist。
3. 当前后台问题盘点。
4. 第一阶段页面范围冻结。

门禁：

1. 不写功能代码。
2. 不允许 agent 自行扩大范围。
3. 后续每个模块必须先有页面目标和验收标准，再实现。

### 阶段 1：后台结构和工作台

交付：

1. 新后台信息架构。
2. 工作台首页。
3. 模块导航重排。
4. 待办卡片和关键异常入口。
5. 通用列表、详情抽屉、确认弹窗、状态徽标、金额展示组件。

优先级：

1. 工作台。
2. 导航。
3. 通用组件。
4. 页面级验收脚本基础。

验收：

1. 登录后第一屏能看到待处理事项。
2. 点击待办能进入对应模块和筛选状态。
3. 不再把所有业务数据混在首页。

### 阶段 2：商品经营重构

交付：

1. 平台商品库页面。
2. 平台自营商品页面。
3. 商户代理商品页面。
4. 自有商品审核和自有商品管理。
5. 卡密池导入、预检、库存、明文查看审计。

验收：

1. 平台运营可以创建商品、设置价格、设置发货方式、上架自营商品。
2. 商户可以从商品列表选择代理商品并修改本店展示内容。
3. 保存代理商品前确认框说明“只修改本店展示，不修改平台商品源”。
4. 自有商品入口命名清楚，不与代理商品混淆。

### 阶段 3：订单履约和售后退款

交付：

1. 订单列表和订单详情抽屉。
2. 待确认收款工作流。
3. 自动发码和人工交付状态展示。
4. 售后单、退款审批、责任拆分、人工退款确认。
5. 退款后卡密禁看。

验收：

1. 订单下一步动作准确。
2. 支付、履约、售后、ledger、audit 状态一致。
3. 重复确认收款、重复退款、重复发码不会造成重复业务结果。

### 阶段 4：商户平台和财务收款

交付：

1. 商户审核和手工创建一级商户。
2. 保证金确认、扣减和流水。
3. 邀请码和下级供货授权。
4. 转供价管理。
5. 收款配置卡片和表单回填。
6. 支付异常、余额充值、服务费、供货应付、追扣、ledger。

验收：

1. M1/M2/M3 登录后菜单和数据范围正确。
2. 未缴保证金不能经营。
3. 价格隔离正确。
4. 收款配置启用后 H5 展示正确，停用后 H5 不展示。
5. 密钥不回显，操作写审计。

### 阶段 5：风控审计和完整页面验收

交付：

1. 风控冻结页面。
2. 审计日志页面。
3. 导出和明文查看权限控制。
4. Playwright 页面级验收。
5. 截图证据归档。

验收：

1. H5、平台后台、M1/M2/M3 商户后台逐页验收。
2. 页面、API、数据库、审计证据一致。
3. P0 阻断项全部清零。

## 7. 第一阶段建议范围

第一阶段建议只做以下内容，避免再次做成“大而乱”的后台：

1. 工作台首页。
2. 后台导航重排。
3. 通用状态徽标。
4. 通用金额展示和输入规则。
5. 通用详情抽屉。
6. 通用危险操作确认弹窗。
7. 商品经营模块的信息架构骨架。
8. 订单履约模块的信息架构骨架。

第一阶段暂不做：

1. 新增真实支付平台联调。
2. 大规模数据库迁移。
3. 新增复杂权限系统。
4. 大量批量操作。
5. 在线客服或工单。
6. 商户提现。

## 8. 页面设计与文案规则

### 8.1 字段命名

必须把工程字段翻译成后台字段：

| 工程含义 | 后台显示 |
| --- | --- |
| `salePriceCents` | 售价 |
| `supplyPriceCents` | 供货价 |
| `minSalePriceCents` | 最低售价 |
| `suggestedSalePriceCents` | 建议售价 |
| `fulfillmentMode` | 发货方式 |
| `platformProductId` | 平台商品 |
| `merchantProductListingId` | 店铺商品 |
| `paymentStatus` | 支付状态 |
| `fulfillmentStatus` | 发货状态 |
| `refundStatus` | 退款状态 |

规则：

1. 页面不展示 `xxxCents`。
2. 页面不要求用户理解 `platformProductId`、`merchantId`、`shopId`。
3. ID 可在详情页“技术信息”折叠区展示，默认隐藏。

### 8.2 表单规则

1. 长表单按业务分组，不超过 4 到 6 个字段一组。
2. 金额输入为元，提示示例为“例如 19.90”。
3. 选择商品、商户、店铺、订单时优先使用搜索列表。
4. 保存前展示变更摘要。
5. 危险操作按钮文案必须说明动作结果，例如“确认到账并发货”“退款并禁看卡密”“冻结店铺”。

### 8.3 列表规则

1. 列表列数控制在 6 到 9 列。
2. 默认列必须包含名称、状态、金额、归属、时间、下一步动作。
3. 次要信息放详情抽屉。
4. 大表必须支持分页、筛选、搜索。
5. 空状态必须告诉用户下一步做什么。

### 8.4 真实操作习惯规则

后台必须按管理员真实操作习惯设计，不能只把接口字段摆成表单。

#### 8.4.1 列表点击与编辑回填

1. 商品、订单、商户、收款方式、优惠券、卡密批次等列表点击后，必须自动带出对应记录的核心字段。
2. 商品列表点击“查看/编辑”后，必须回填商品图片、商品名称、类目、标签、详情、使用说明、发货方式、库存、供货价、最低售价、建议售价、状态等核心字段。
3. 平台自营商品点击后，必须回填平台自营售价、履约成本、销售状态和关联平台商品，不要求管理员手填平台商品 ID。
4. 收款方式点击“配置/查看”后，必须回填当前方式的展示名称、确认方式、是否默认、启用状态、账号标识、二维码/支付链接、密钥配置状态和运营备注。
5. 未选择记录时，编辑区必须明确提示“先从列表选择一条记录”，不能展示一堆空表单让管理员猜。

#### 8.4.2 修改前后对比确认

1. 所有会写入数据库的保存动作，都必须先弹出二次确认窗口。
2. 确认窗口必须展示“修改前”和“修改后”的核心字段对比。
3. 商品修改至少展示：商品名称、图片、发货方式、库存、供货价、最低售价、建议售价、状态。
4. 收款配置修改至少展示：收款方式、展示名称、确认方式、启用状态、默认状态、密钥状态、二维码/支付链接是否变更。
5. 保证金、退款、结算、冻结、卡密明文查看等高风险动作必须展示影响结果，例如金额变化、订单状态变化、是否禁看卡密、是否写审计。
6. 管理员在确认窗口点击“确认修改/确认执行”后，才正式提交 API。
7. 确认窗口必须有明确取消入口；取消后不改变数据。

#### 8.4.3 导入导出模板

1. 所有导入功能都必须提供“下载模板”按钮。
2. 模板必须包含字段名、示例值、必填说明和格式说明。
3. 卡密导入模板必须说明：一行一个卡密、可选批次号、禁止重复、空行处理、非法字符规则。
4. 商品批量导入模板必须说明：商品名称、类目、发货方式、金额单位、库存、状态、详情字段格式。
5. 收款配置、优惠券、商户导入如后续支持批量导入，也必须先有模板。
6. 导入前必须有预检步骤，展示总行数、有效行、重复行、非法行、空行和错误原因。
7. 导入后必须展示导入结果：成功数量、失败数量、失败原因、可下载失败明细。
8. 导出必须说明导出范围和脱敏规则，高风险导出写审计。

#### 8.4.4 按钮点击保护与幂等

1. 所有提交按钮点击后必须进入 loading 状态，并在请求完成前禁用。
2. 同一动作不能因为连续点击产生重复创建、重复确认、重复发码、重复退款、重复结算或重复导入。
3. 前端按钮禁用只是第一层保护；后端 API 必须通过幂等键、唯一约束、事务或状态机防重复。
4. 创建、保存、确认到账、发货、退款、生成结算单、确认打款、导入卡密、查看明文等动作必须有处理中提示。
5. 请求失败后必须展示具体错误，并允许管理员按当前数据状态重新操作。
6. 如果动作已经被其他管理员处理，页面必须提示“状态已变化，请刷新后再操作”，不能继续提交旧状态。

#### 8.4.5 卡密库存与分配规则

1. 自动发码商品必须在合理位置展示卡密库存：可用数量、已分配数量、已使用数量、冻结数量、退款禁看数量、失效数量。
2. 商品详情页必须能进入该商品的卡密池，不要求管理员另找入口。
3. 卡密池必须按商品、批次、状态筛选。
4. 卡密列表默认脱敏展示；具备权限的管理员查看明文必须二次确认并写审计。
5. 一个商品有多条卡密时，系统必须明确分配规则，例如按导入时间 FIFO 分配，或按批次优先级分配；规则必须写在页面说明和后端逻辑里。
6. 订单确认收款后，自动卡密商品只能分配未使用、未冻结、未失效且归属正确的卡密。
7. 并发购买最后一张卡密时，只能有一个订单分配成功，其余订单必须进入库存不足或待处理状态。
8. 已分配卡密必须能从订单详情看到脱敏信息，并能回溯到商品、批次、订单和分配时间。
9. 退款后买家不能继续查看卡密，但后台可按权限和审计查看历史分配记录。

#### 8.4.6 收款配置表单说明

收款配置不能是一堆无解释的 textbox。必须按收款类型动态展示字段，并说明哪些必填、哪些选填、填错会影响什么。

1. 选择“个人支付宝/个人微信”时，只展示个人码需要的字段：展示名称、二维码、支付说明、是否启用、是否默认、运营备注。
2. 选择“支付宝商户”时，展示商户号、AppID、私钥/公钥/证书、回调地址、查单配置、签名测试入口，并说明密钥不回显。
3. 选择“微信/腾讯商户”时，展示商户号、AppID、服务商号、证书、API 密钥、回调地址、查单配置、签名测试入口。
4. 选择“e支付”时，展示服务商商户号、网关地址、签名密钥、支付链接/接口模式、回调地址和查单配置。
5. 选择“余额支付”时，只展示余额支付是否启用、充值确认方式和手续费说明，不展示无意义密钥字段。
6. 每个字段旁必须有短说明或 placeholder，说明填写来源和用途。
7. 保存前确认窗口必须说明该收款方式会影响哪个店铺、哪些订单支付入口和 H5 展示。
8. 配置不完整时不能启用，只能保存草稿或提示缺少哪些字段。

## 10. 开发协作规则

后续 agent 或开发者执行时必须遵守：

1. 先读本文，再改后台。
2. 每次只改一个模块或一个跨模块基础组件。
3. 不允许把新功能继续堆回单个超大文件。
4. 不允许把数据库字段直接暴露成表单字段。

## 11. 建议优先级

P0：

1. 工作台待办和异常入口。
2. 商品经营任务流。
3. 订单支付/履约下一步动作。
4. 商户保证金门槛和权限裁剪。
5. 收款配置不回显密钥、启停和 H5 联动。
6. 卡密导入、发码、退款后禁看。
7. 售后退款责任拆分和幂等。
8. 商品列表点击后自动回填核心字段，保存前展示修改前后对比确认。
9. 所有导入功能提供模板下载、预检和失败明细。
10. 所有写操作按钮具备点击保护，后端具备幂等保护。
11. 卡密池展示库存、批次、状态、分配记录和明确分配规则。
12. 收款配置按类型动态展示字段和填写说明，不再展示一堆无解释输入框。

P1：

1. 财务对账、服务费、供货应付、追扣页面。
2. 风控冻结和审计日志体验。
3. 大表分页、筛选、搜索。
4. 商户下级供货授权和转供价管理体验。

P2：

1. 批量审核、批量上下架、批量导出。
2. 更完整的经营看板。
3. 邮件服务配置和投递记录。
4. 更细的后台角色权限管理。
