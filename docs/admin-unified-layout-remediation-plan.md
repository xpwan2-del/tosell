# 后台统一布局整改方案

本方案用于把 ToSell 后台从“局部堆功能”改成可长期维护的运营后台。后续商品、库存、订单、售后、财务和配置页面都必须按这里的布局和组件规则执行。

## 1. 整体布局标准

### 1.1 后台骨架

后台固定使用三层结构：

1. 左侧导航：只负责模块入口和分组，不放业务数据。
2. 顶部栏：只展示当前后台身份、当前模块上下文、刷新、退出和全局状态。
3. 内容区：承载模块标题、操作区、列表、详情和弹窗。

禁止每个页面自己发明一套顶部、标题、说明和按钮排法。

### 1.2 左侧导航

左侧导航按业务域分组：

1. 总览：首页。
2. 经营：商品、自动发货库存、优惠券、订单、发货、售后。
3. 数据：销售统计。
4. 店铺：店铺设置。
5. 平台：商户管理、下级供货、风控日志。
6. 财务：结算、收款配置。

规则：

1. 导航文字必须短，优先 2 到 6 个字。
2. 当前模块高亮必须明显。
3. 手机端左侧导航必须收起为顶部模块菜单或抽屉，不挤占内容宽度。
4. 不允许在导航里显示长说明、数字解释或临时提示。

### 1.3 顶部栏

顶部栏固定展示：

1. 当前后台名称。
2. 当前账号角色。
3. 全局刷新按钮。
4. 退出按钮。
5. 全局消息提示。

规则：

1. 顶部栏不放页面级搜索。
2. 顶部栏不放当前表单按钮。
3. 成功、失败、处理中提示统一走全局消息条，不在多个位置重复提示。

### 1.4 内容区和模块标题

每个模块顶部必须统一：

```text
模块标题
一句话说明当前模块处理什么
怎么用：最多 2 条，必须是人的操作步骤
```

规则：

1. 模块标题使用页面级字号。
2. 页面说明使用普通正文，不用大字。
3. “怎么用”只写操作顺序，不写技术接口、RBAC、审计表名。
4. 每个模块只允许一个主操作区，辅助操作放到详情或批量工具栏。

## 2. 页面类型标准

### 2.1 列表页

适用：商品库、订单、售后、商户、结算、库存明细、收款方式列表。

标准结构：

```text
模块标题
搜索区
批量/主要操作区
列表
分页
详情抽屉或详情页
```

规则：

1. 搜索区必须在列表上方。
2. 搜索区必须有“搜索”和“重置”按钮。
3. 关键筛选字段必须拆开，不用一个万能输入框糊弄。
4. 表格只放关键字段，详情放抽屉。
5. 操作列固定在最左或最右，同一项目保持一致。
6. 列表为空时显示“为什么为空”和“下一步做什么”。

### 2.2 详情页 / 详情抽屉

适用：商品详情、订单详情、售后详情、商户详情、收款方式详情。

标准结构：

```text
详情头部：对象名称、状态、关闭/保存
左侧：图片或核心摘要
中间：可编辑基础信息
右侧或下方：只读概况、审计、关联对象
底部：保存/取消/高风险操作
```

规则：

1. 详情页必须先展示核心身份：名称、编号、状态、归属。
2. 可编辑字段和只读信息必须分区。
3. 高风险操作必须二次确认。
4. 自动发货库存这类子业务只在详情里显示概况，不在详情里做批量导入。
5. 抽屉宽度和字体层级统一，禁止每个抽屉自己改字号。

### 2.3 表单页

适用：新增商品、编辑收款方式、创建优惠券、添加库存。

标准结构：

```text
表单说明
分组字段
字段解释
操作按钮
保存前确认
```

规则：

1. 字段必须按业务分组，不允许一堆 textbox 直接堆上去。
2. 必填字段标注在 label 或 placeholder，不写在远处说明。
3. 字段名必须是运营能懂的中文。
4. 技术字段和内部 ID 默认只读或折叠。
5. 保存按钮必须有 loading 和防重复提交。

### 2.4 审核页

适用：商户审核、自有商品审核、收款配置审核、退款审批。

标准结构：

```text
待审核列表
审核详情
审核历史
通过 / 拒绝 / 关闭
```

规则：

1. 审核列表默认优先显示待处理。
2. 已审核记录可以查，但不能和待审核混成一团。
3. 审核详情必须展示提交人、提交时间、审核人、审核时间、审核结论。
4. 通过和拒绝必须二次确认。
5. 拒绝必须填写原因。

### 2.5 自动发货库存页

适用：兑换码、卡密、账号密码库存管理。

标准结构：

```text
自动发货商品列表
当前商品库存管理
添加库存
库存明细
敏感内容查看审计
```

规则：

1. 库存管理页才允许导入兑换码、卡密或账号密码。
2. 商品详情页只显示自动发货概况和“去库存管理补货”。
3. 当前库存明细默认只显示当前商品。
4. 默认只显示内容预览。
5. 查看完整库存内容、下载完整库存表必须二次确认并写审计。
6. 兑换码/卡密和账号密码共用同一套自动发货库存模型，但导入模板和输入提示必须按凭证类型变化。

### 2.6 财务页

适用：收款配置、充值、退款、结算、保证金。

标准结构：

```text
财务对象列表
当前对象详情
金额/状态/凭证
确认操作
审计和流水
```

规则：

1. 金额必须用元展示，输入可注明分或元，但同一页面只能选一种主表达。
2. 所有确认到账、退款成功、打款成功都必须二次确认。
3. 财务页必须展示凭证、操作人、操作时间和状态变化。
4. 技术接口、回调、密钥状态放在折叠详情，不占主流程。

## 3. 组件标准

### 3.1 搜索区

统一使用结构化搜索：

1. 第一行说明可搜索什么。
2. 字段按业务拆开。
3. 最后一组固定为“搜索 / 重置”。
4. 搜索结果数量显示在搜索区底部。

禁止：

1. 只有一个“筛选当前表格”的输入框。
2. 搜索字段没有提示。
3. 状态、价格、日期全部塞进关键词。

### 3.2 表格

表格用于桌面端高密度信息。

规则：

1. 表头必须是人话中文。
2. 金额统一格式化。
3. 状态必须使用状态标签。
4. 长文本截断，详情抽屉展示完整内容。
5. 表格默认分页。
6. 手机端不直接横向塞完整表格，改成卡片列表。

### 3.3 卡片

卡片只用于对象列表或摘要，不用于套娃装饰。

适用：

1. 商品卡。
2. 自动发货商品库存卡。
3. 订单摘要卡。
4. 财务待办卡。

规则：

1. 卡片必须有明确对象名称。
2. 卡片只放 3 到 6 个关键事实。
3. 操作按钮放在卡片右侧或底部，保持统一。
4. 不允许卡片里再放大卡片。

### 3.4 抽屉

抽屉用于详情和编辑。

规则：

1. 抽屉标题、关闭按钮位置固定。
2. 抽屉内 tab 不超过 4 个。
3. 主编辑区和概况区分开。
4. 保存按钮在固定操作区。
5. 手机端抽屉变全屏页面。

### 3.5 弹窗

弹窗只用于确认、危险操作和短流程，不做长表单。

规则：

1. 标题写清楚操作对象。
2. 正文展示修改前和修改后。
3. 确认按钮写具体动作，例如“确认修改”“确认退款成功”。
4. 取消按钮永远在确认按钮旁边。
5. 弹窗内错误必须显示在按钮上方，不跳到页面顶部。

### 3.6 按钮

按钮分三类：

1. 主按钮：保存、提交、确认、添加库存。
2. 次按钮：取消、关闭、重置、下载模板。
3. 危险按钮：拒绝、退款成功确认、删除、冻结。

规则：

1. 同一操作区只能有一个主按钮。
2. 按钮文案必须是动作，不写抽象名词。
3. 点击后必须 loading 和 disabled。
4. 高风险按钮必须二次确认。

### 3.7 状态标签

状态标签必须统一颜色语义：

1. 正常/通过/启用：绿色。
2. 待处理/处理中：黄色。
3. 失败/拒绝/异常：红色。
4. 冻结/停用：灰色。
5. 信息/只读：蓝绿色。

状态文案必须是中文业务状态，不直接展示后端枚举。

### 3.8 提示文案

提示文案只回答三件事：

1. 这个区域是干什么的。
2. 管理员下一步该做什么。
3. 操作后会影响什么。

禁止在页面正文里展示技术实现，例如接口名、RBAC 校验、数据库表名、审计 action 名。技术信息只能放到审计或开发调试区。

## 4. 移动端标准

后台不是所有页面都适合手机完整维护。移动端按能力分级。

### 4.1 手机可维护

这些页面必须支持手机操作：

1. 首页待办。
2. 订单查看和处理。
3. 售后查看和基础处理。
4. 发货确认。
5. 收款到账确认。
6. 库存数量查看。

要求：

1. 表格变卡片。
2. 操作按钮固定在卡片底部。
3. 抽屉变全屏。
4. 搜索区默认折叠，只展示关键搜索。
5. 金额、订单号、状态必须清晰可读。

### 4.2 手机只读或轻维护

这些页面手机端以查看为主：

1. 商品详情。
2. 自动发货库存列表。
3. 商户详情。
4. 结算详情。
5. 销售统计。

允许的手机操作：

1. 查看状态。
2. 查看库存数量。
3. 搜索。
4. 跳转详情。

不建议手机操作：

1. 大批量导入卡密。
2. 大段编辑商品详情。
3. 复杂财务拆账。
4. 批量导出完整敏感数据。

### 4.3 手机不提供主流程

这些操作默认要求桌面端：

1. 批量导入库存。
2. 下载完整库存表。
3. 商品详情长表单编辑。
4. 复杂结算生成。
5. 多条件财务对账。

手机端看到这些操作时，应提示“请在桌面端处理”，不能硬塞一个难用表单。

## 5. 改造顺序

### 阶段 0：建立规则和验收方式

目标：先定标准，不再散改。

任务：

1. 把本方案纳入后台改造基准。
2. AGENTS 增加后台设计系统规则。
3. Browser 检查桌面、平板、手机三个视口。
4. 明确功能验收必须走 Google VM PostgreSQL 真库，视觉检查可以本地完成。

验收：

1. 有统一文档。
2. 有统一规则。
3. 后续改造不再新增散乱页面模式。

### 阶段 1：后台骨架和基础组件

目标：先统一壳子。

任务：

1. 统一左侧导航、顶部栏、内容区宽度和间距。
2. 统一 Module、Panel、Table、Search、Drawer、ConfirmModal。
3. 整理 CSS token：字体、颜色、间距、边框、阴影、状态色。
4. 禁止页面级随意写新字号、新间距、新颜色。

验收：

1. 各模块标题、说明、按钮位置统一。
2. 手机端导航不挤压内容。
3. 主要组件在 375、390、430、768、1440 宽度不溢出。

### 阶段 2：商品和自动发货库存

目标：先修当前最乱的核心经营区。

任务：

1. 商品列表统一搜索、卡片、详情抽屉。
2. 商品详情只做商品设置和自动发货概况。
3. 自动发货库存页只做库存管理。
4. 账号密码、兑换码/卡密导入模板和提示统一。
5. 商品审核独立成审核列表和审核详情。

验收：

1. 管理员知道商品在哪改、库存在哪补。
2. 商品详情不再出现库存导入表单。
3. 库存页按当前商品展示库存。
4. 手机端商品页可查看，复杂编辑提示桌面端。

### 阶段 3：订单、发货、售后

目标：把经营闭环改成人能操作的工作台。

任务：

1. 订单列表支持订单号、买家、店铺、商品、支付状态、发货状态、退款状态搜索。
2. 订单详情展示订单金额、发货凭证、支付方式、商户链路、审计。
3. 发货页区分自动发货概况和人工交付待处理。
4. 售后页改成订单/售后列表 -> 售后详情 -> 拆账 -> 审批 -> 退款确认。
5. 退款后禁看库存凭证的状态必须展示清楚。

验收：

1. 售后页面不是空表单，而是从订单或售后单进入处理。
2. 每个退款动作能看到金额、责任方、凭证和状态。
3. 手机端至少能查看订单和售后详情、处理基础确认。

### 阶段 4：财务和配置

目标：让收款、结算、保证金、充值不再像技术配置表。

任务：

1. 收款配置按“个人码 / 官方支付 / e支付 / 余额”分组。
2. 每种收款方式只显示相关字段。
3. 密钥、回调、查单结果放技术详情折叠区。
4. 结算页按待生成、待确认、已完成、异常分区。
5. 保证金和充值统一金额、凭证、操作人、状态流。

验收：

1. 添加收款方式时字段解释清楚。
2. 管理员知道哪些字段必填、哪些不用填。
3. 财务确认动作都有二次确认和审计。

### 阶段 5：全后台移动端整改

目标：把手机端从“勉强打开”改成“能处理紧急事项”。

任务：

1. 所有表格移动端卡片化。
2. 搜索区移动端折叠。
3. 抽屉移动端全屏。
4. 大表单移动端提示桌面处理。
5. 核心待办手机端可点、可确认、可查看详情。

验收：

1. 375px 宽度不横向溢出。
2. 390px 和 430px 常见手机宽度可读可点。
3. 订单、售后、发货、到账确认可在手机完成基础处理。

## 6. 禁止事项

1. 禁止局部页面自己写一套布局。
2. 禁止一个 panel 里塞多个互不相关的业务。
3. 禁止技术说明占据运营主界面。
4. 禁止万能搜索框冒充组合搜索。
5. 禁止商品详情、库存管理、发货管理职责混在一起。
6. 禁止手机端直接塞完整桌面表格。
7. 禁止只看本地内存测试就说功能通过。

## 7. 后续执行方式

每次改造一个模块时，必须按以下顺序：

1. 先画当前模块的信息结构。
2. 再确认页面类型：列表、详情、表单、审核、库存、财务。
3. 按本方案套组件标准。
4. 改完做桌面、平板、手机 Browser 视觉检查。
5. 涉及业务写入时，走真实数据库验收。
6. 最后更新文档和验收记录。
