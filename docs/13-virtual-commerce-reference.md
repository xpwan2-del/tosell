# 虚拟商城竞品借鉴与三级代理口径

状态：行业/竞品参考资料，不作为开发、测试或验收依据。若本文与产品需求、业务规则、收款绑定、页面级验收文档冲突，以 `docs/01-product-requirements.md`、`docs/02-business-rules.md`、`docs/16-page-level-production-acceptance-test-plan.md`、`docs/18-payment-binding-and-voucher-removal-plan.md` 为准。

## 1. 正确对比口径

ToSell 与 `liyaohui.xyz` 同属虚拟商品销售场景，核心商品形态都以虚拟账号、卡密、会员权益、兑换码和人工交付服务为主。差异不在于是否销售虚拟产品，而在于经营组织模型。

`liyaohui.xyz` 更像单店或单平台虚拟商品商城，强项是前台成交路径直接：分类、商品卡、价格、库存、销量、自动发货、登录注册、订单查询、购买须知和客服入口都很清楚。

ToSell 要在同样的虚拟商品前台成交体验之上，增加多商户、多店铺、最多三级代理价差供货、独立客服、后台审核、库存归属、订单归属、售后责任、结算和权限隔离。

一句话：前台成交体验要向成熟虚拟商品商城学习；后台和数据模型必须支撑最多三级代理价差供货。

## 2. 前台 P0 必须借鉴

1. 商品分类清晰：热门、会员权益、账号卡密、自动发货、人工发货、最新上架。
2. 商品卡直接成交：商品名、价格、库存、销量、发货方式、标签、立即购买。
3. 商品详情完整：权益说明、使用说明、库存状态、发货规则、售后规则、客服入口。
4. 买家入口完整：登录/注册、订单查询、订单详情、售后状态。
5. 购买须知明确：虚拟商品规则、发货时间、售后条件、支付确认和购买密码说明。
6. 客服入口高可见：店铺页、商品页、订单页、售后页均可触达店铺客服二维码。
7. 自动发货体验完整：启用购买密码的商品确认收款后进入提取页，输入购买时设置的购买密码查看卡密、账号或兑换码；未启用购买密码的商品在订单详情按权限查看。
8. 人工发货体验清楚：订单显示待处理、处理中、已发货和客服联系方式。
9. 移动端优先：首屏直接展示商品和购买按钮，不用先解释平台模式。

## 3. ToSell 额外 P0 能力

### 3.1 供货链模型

允许的最长链路：

```text
平台供货 -> 一级代理 -> 二级代理 -> 三级代理 -> 消费者
```

规则：

1. 最多三级代理。
2. 三级代理只能销售，不能继续创建四级。
3. 每个订单必须固化完整供货链快照。
4. 各级收益来自供货价差和转供价差。
5. 禁止返佣、分佣、团队奖、邀请奖励和拉人头收益。

### 3.2 价格规则

1. 平台设置平台供货价、最低限价、建议价。
2. 一级代理设置给二级代理的一级转供价。
3. 二级代理设置给三级代理的二级转供价。
4. 二级或三级代理设置最终销售价。
5. 一级转供价不得低于平台供货价。
6. 二级转供价不得低于一级转供价。
7. 最终销售价不得低于销售代理的上游供货价和平台最低限价。
8. 文案统一使用：供货价差、转供价、经销价差。

### 3.3 每层店铺能力

一级、二级、三级代理都必须具备：

1. 独立店铺页。
2. 独立客服二维码。
3. 独立商品列表。
4. 独立商品定价。
5. 独立订单归属。
6. 独立售后责任。
7. 独立供货清算和服务费对账记录。
8. 独立销售统计。

三级代理不具备：

1. 新增下级代理。
2. 配置给下级的转供价。
3. 创建四级渠道或四级店铺。

### 3.4 后台权限

平台后台：

1. 查看完整供货链。
2. 审核一级、二级、三级代理。
3. 管理全平台商品、库存、订单、售后、供货清算、审计和风控。
4. 冻结店铺、代理、订单和供货清算。

一级代理后台：

1. 管理自己的店铺和订单。
2. 管理自己下游二级和对应三级链路。
3. 配置给二级的转供价。
4. 查看自己的一级价差、供货清算和追扣。

二级代理后台：

1. 管理自己的店铺和订单。
2. 管理自己下游三级链路。
3. 配置给三级的转供价。
4. 查看自己的二级价差、供货清算和追扣。

三级代理后台：

1. 管理自己的店铺、商品售价、订单、发货、售后和供货清算。
2. 不显示新增下级、下级管理和转供价给下级功能。

## 4. 库存、售后和供货清算

库存必须记录来源：平台库存、代理自有库存、上级分配库存。卡密或账号发放必须记录库存来源、销售店铺、订单号和责任主体。自动发货必须避免并发重复发码。

售后责任至少覆盖：平台、一级、二级、三级、销售店铺、买家和混合责任。售后责任影响退款、补偿、追扣和供货清算冻结。

供货清算必须按供货价差生成。清算单记录订单、角色、层级、供货价、转供价、销售价、价差、服务费、追扣和清算金额。商户使用自己的收款码/链接收款，P0 不做平台代收后 T+1 给商户打款；清算必须排除售后中、履约失败、库存异常和风控冻结订单。

## 5. 不能照搬

1. 不能照搬单店后台结构，ToSell 必须有平台后台、商户后台和代理后台。
2. 不能照搬高风险商品品类，商品类目必须符合 H5 平台运营、支付方式和相关合规要求。
3. 不能让库存为 0 的商品继续可购买。
4. 不能把客服 QQ 固定为平台客服，每个店铺必须能配置自己的客服二维码。
5. 不能使用返佣、分佣、团队奖励、邀请奖励、代理等级等文案。
6. 不能只做本地订单缓存，订单和卡密必须由服务端持久化。
7. 不能把商品、图片、详情、价格、库存、卡密、客服二维码、收款码或支付链接硬编码在前端。
8. 不能把后台操作固定到测试店铺、测试商户、测试商品或测试订单。

## 6. 更新后的验收标准

1. 前台达到虚拟商品商城基础体验：分类、商品卡、价格、库存、销量、发货方式、登录、订单查询、客服、售后。
2. 平台自营店、一级店、二级店、三级店都能独立打开和销售。
3. 每个代理都有自己的店铺页、客服二维码、收款说明、商品定价、订单、售后和供货清算。
4. 平台能审核一级、二级、三级代理和保证金。
5. 一级能管理自己下游二级及其三级链路。
6. 二级能管理自己下游三级链路。
7. 三级不能创建四级。
8. 每级转供价校验正确。
9. 订单快照固化完整供货链、每级价格和每级经销价差。
10. 自动发货不重复发码，人工发货有记录。
11. 售后责任能分摊到平台、一级、二级、三级、销售店铺或买家。
12. 供货清算按供货价差生成，不出现返佣、分佣、团队奖、邀请奖励口径。
13. 支付最后接；非支付阶段先用商户收款配置和系统确认收款闭环，mock 只能用于开发测试，不能成为生产页面入口。
14. 生产运行代码不出现固定测试店铺、固定测试商品、固定收款码或固定 mock 支付。
